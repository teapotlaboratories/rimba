Rimba regression suite — the complete guide

What the suite is, why it exists, what each test proves (and deliberately does not), and the exact hardware each needs. The companion to the terse how-to (tools/regtest/README.md) and the living results (rimba-regression-results.md).

1 · The goal — why this exists

Rimba is ESP32-S3 firmware on a vendored, moving HaLow stack (MorseMicro's morselib + a hostap shim). Firmware blobs, morselib, and ESP-IDF all get bumped, and much of Rimba is ported from the Linux reference. Every bump risks silently regressing a milestone that worked yesterday — a dead radio, a broken mesh, a doubled power-save current. The suite makes "is everything that worked yesterday still working?" a runnable answer instead of a bench session of rediscovery.

The core principle: honest tiers

Every test is explicit about what it proves and what it does NOT. A test that overclaims is worse than no test — it gets trusted. So the suite is split by how much hardware it needs and how strong a claim it supports: cheap-and-weak first, expensive-and-strong last.

T0 also enforces two build-time invariants no other tier can: the country code (a "??" from a bare idf.py build makes morselib refuse the radio — looking exactly like dead hardware), and the fw-blob version-pin — mm6108.bin must be the 1.17.8 blob by size 480664 B + sha256 (a silent bump to 1.17.9 ~doubles STA power-save current). T1 reads values not line-presence: country=US chip=0x0306 fw=1.17.8 + a real MAC — a dead radio prints an uninitialised struct, which T1 rejects.

4 · How a T2 test runs

Every T2 app is a self-contained test-* app reporting over the TEST| console contract — one format, both a human and the scraper read it:

TEST|BEGIN|name=<slug>|rig=<what this needs>
TEST|INFO|<progress / observed values>
TEST|STEP|<step>|<PASS|FAIL>|<detail>      ← a sub-check
TEST|RESULT|<PASS|FAIL|INCONCLUSIVE>|<detail>
TEST|END|name=<slug>

6 · The tp power-save tier

board2 runs a fixed 4-tier current ladder; the C6 triggers it; the host samples the PPK2 and segments the current stream by the firmware's TEST| phase markers. A power verdict is host-side (the firmware can't see the current), so it's a separate tier.

board2 · PPK2 ampere-meter · median mA · dashed = calibrated FAIL band

4 measured, 2 scored. No-PS (~63 mA) is a validity check (associated + PS on + not overloaded); Dyn-PS + WNM+chip-powerdown are the scored discriminators; TWT is recorded (a Linux AP ignores the mid-session setup). Wide gross-multiple gate — doze depth is a benchmark, so PASS ≤ ~1.4× the calibrated baseline, FAIL ≥ ~1.8–2×, INCONCLUSIVE between, raw mA always recorded. tp --light-sleep builds a stronger variant (separate bands).

Known offset — NOT an AP-TX / overload issue

This rig reads ~1.6–2× the documented reference on the doze tiers. Verified not overload: both APs + the DUT are already TX-capped to ~0–1 dBm (uint16_t min 0), both AP paths read equal, and the link is healthy (No-PS ~63 mA; overload reads wrong ~14). It's an open measurement/config question — the bands are calibrated to this rig, so the gate is valid.

7 · Status vocabulary

Richer than pass/fail on purpose — conflating an honest non-result with a regression is how a suite gets ignored. Exit is 0 only when nothing FAILed (SKIP / INCONCLUSIVE / XFAIL don't gate).

StatusMeaningGates?
PASS / FAILas expectedFAIL gates
SKIPcould not run (board absent, not implemented)no
INCONCLUSIVEran, but the measurement can't be trusted (noisy RF, flaky link) — not a code regressionno
XFAILfailed for a documented reason (proto1's missing BCF)no
XPASSa known-broken case unexpectedly passed → the reason is staleyes

8 · Running it

source vendor/esp-idf/export.sh          # once per shell (anything touching a board)
make test-bench                          # what hardware is present right now

make test-t0                             # build matrix (no hardware)
make test-t1                             # smoke on board2 (radio boots)
make test-t2                             # all 12 on-air feature tests
make test-tp                             # PPK2 power-save ladder
python tools/regtest/run.py dscycle      # deep-sleep reconnect gate
python tools/regtest/run.py tp --light-sleep   # the stronger light-sleep variant
python tools/regtest/run.py t2 --dry-run       # the T2 catalogue, no hardware

make test-report                         # (re)generate build/regtest/report.html
make test-silence                        # return every ESP to the idle app

Each run writes a git-stamped JSON baseline (build/regtest/<tier>-latest.json — a regression is a JSON diff) and refreshes a self-contained report.html. Long runs accumulate across kill-safe chunks with --append. Start tools/ppk2_hold.py before anything using board2; recover a deep-sleep-wedged board2 with a 5 s PPK2 power-off (2.5 s leaves it dark) + a tight esptool flash of test-idle.
