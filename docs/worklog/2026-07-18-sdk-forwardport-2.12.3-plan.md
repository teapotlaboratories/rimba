# SDK forward-port plan — mm-iot-sdk 2.10.4 → 2.12.3 (into our repo)

**2026-07-18 · planning (grounded in a systematic per-file analysis + direct git verification). NOT yet applied.**

## Goal & locked decisions
Bring MorseMicro's upstream SDK **into our repo** — upgrade the `components/halow` submodule
(`teapotlaboratories/mm-esp32-halow`) from **morselib 2.10.4 → 2.12.3**, keeping our self-contained
vendored layout + our 24 merged PRs, tagging each milestone. Owner-approved decisions:
- **Staged** `2.10.4 → 2.11.2 → 2.12.3`, compile-verified + **tagged** at each hop (`-esp32-1`).
- **Tag style** `-esp32-N` (e.g. `2.12.3-esp32-1`) — honest about "upstream + our patches".
- **Ongoing tracking** — keep the `mm` remote (`MorseMicro/mm-iot-sdk`) + this repeatable procedure.
- **Mechanics: Option B** — 3-way diff-apply onto our tree, `095b1290`(2.10.4) as the near-exact base.

Work branch: `feat/sdk-forward-port-2.12.3` (off submodule `main` @ `3c8ded0d`). Commits allowed
Saturdays; owner-attributed; branch+PR + **/review before merge**; **re-run the full regression suite**
after (bench radio-silent after tests).

## Layout & the SCOPE decision (critical)
Our ESP build compiles the SDK **from source** at `components/mm-iot-sdk/framework/…` via *explicit*
source lists in each ESP-component `CMakeLists.txt` (`MM_IOT_SDK = components/halow/components/mm-iot-sdk`):

| ESP component | upstream source root it compiles |
|---|---|
| morselib | `framework/morselib` |
| hostap   | `framework/src/hostap` |
| mmpktmem | `framework/src/mmpktmem` |
| mmutils  | `framework/src/mmutils` |
| regdb    | `framework/src/mmregdb` |
| (morselib build tooling) | `framework/tools/{metadata/protected_syms.txt, buildsystem/librarymangler.py}` |
| shims    | ESP-local (`components/shims/`) — no upstream source |

The full upstream `framework/` diff is **mostly irrelevant to us**: hop1 touches ~4058 files but 1259 are
`doc/html`, 832 `freertos-libs`, 690 `aws`, 461 `bsps`, plus STM32 `platforms`, `mbedtls`, `mmagic`,
`examples` — **none of which our ESP build compiles or can compile-verify**. Therefore the forward-port is
**scoped to the compiled roots above**; uncompiled vendored dirs stay at 2.10.4 (dead weight, harmless).
This is consistent with the `-esp32-N` tag choice (not a pristine upstream mirror). The alternative — a full
2.12.3 mirror of the entire vendored tree — pulls in thousands of unverifiable file changes for zero ESP
benefit. **>> OWNER TO CONFIRM: compiled-scope (recommended) vs full-tree mirror. <<**

## Two crucial corrections from the analysis (do not trust the old memory verbatim)
1. **All hard breaking changes are in HOP 1 (2.10.4→2.11.2)**, not spread across hops. This was
   adversarially verified: at 2.11.2 the datapath STA/AP split, `umac_interface_add(datapath_ops,…)`,
   `morse_cmd_tx(…,skip_errlog)`, and the reshaped `umac_datapath_ops` struct are **already in place and
   unchanged at 2.12.3**. **Hop 2 is almost entirely additive** (relay subsystem, DPP QR-code).
2. `construct_80211_data_header` and `frames_allowed_pre_association` are **RETAINED** (memory said
   dropped). `umac_interface_type_is_compatible_with_active` **does not exist at either endpoint**.

## Conflict surface (git ground-truth, scoped to compiled roots)
- **Hop 1**: 402 upstream file changes → **31 conflicts** (30 `morselib`, 1 `hostap/wpa_supplicant/ap.c`).
  `protected_syms.txt` is conflict-free (upstream didn't touch it).
- **Hop 2**: 44 upstream file changes → **11 conflicts** (10 `morselib`, 1 `hostap/hostap_morse_common.h`).
- The **268-file hostap (wpa_supplicant) update in hop1** applies to us but overlaps our edits on only
  `ap.c`. It is the main **BENCH-RISK** area: our mesh SAE/AMPE security integrates with hostap, and

## Mechanics (per hop)
```
# one-time: real upstream remote (done)
git remote add mm git@github.com:MorseMicro/mm-iot-sdk.git ; git fetch mm --tags
ROOTS="framework/morselib framework/src/hostap framework/src/mmpktmem framework/src/mmutils framework/src/mmregdb framework/tools"
# hop1 (095b1290->b61d4548); hop2 (b61d4548->88606f94):
git -C components/halow diff <from> <to> -- $ROOTS   | git -C components/halow apply --3way --directory=components/mm-iot-sdk
# then, MANUALLY (git apply won't): sync each ESP component CMakeLists.txt SRCS list for
#   upstream-ADDED files (umac_datapath_sta.c, umac/relay/*, umac/health_check/*, coredump.c,
#   firmware_mbin.c, …) and upstream-DELETED files (driver/health/driver_health.c, …).
# resolve the conflicts below -> compile-verify -> grep must-survive -> commit -> tag <ver>-esp32-1
```
**CMakeLists source-list sync is a first-class step** — new upstream `.c` files won't build (and their
symbols won't link) until added to the right `components/*/CMakeLists.txt`; deleted ones must be removed.
Compile-verify (link stage) is the safety net that catches a missed sync.

## Compile-verify
Harness: `make build APP=<app> BOARD=proto1-fgh100m` (vendored ESP-IDF 5.4.2). Baseline `rimba-halow-mesh`

`feat/mesh-ap-concurrency` — are skipped here; re-verified separately later.)
