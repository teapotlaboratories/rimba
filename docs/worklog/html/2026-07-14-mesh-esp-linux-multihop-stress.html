ESP↔Linux 3-hop mesh stress
board0 → board1 → board2 → chronite — plus a clean counter-example to the #20 "universal FW limitation"

A forced 3-hop line from an ESP origin, through two ESP relays, to a HW-crypto Linux endpoint (chronite), stress-tested for 30 min one-directional and 20 min bidirectional. Delivery held ~99–100% both ways with zero crashes and no heap leak. Encryption verified on all four nodes. Two findings: a HW-crypto Linux node received multi-hop A4≠TA forwards fine (contradicting #20), and an observed board0 pinger stall was TX saturation, not a mesh fault. No firmware change — the defrag-before-decrypt fix was untouched.

Setup

  • The morselib allowlist gates both peering (umac_mesh.c:1659/2820) and HWMP (:2362) — so the return path chronite→board0 is forced via board2→board1→board0 (mpath HOP_COUNT 3), single-route, no flap. Set before mmwlan_mesh_start (app_main.c:271).
  • Secured SAE+AMPE+CCMP, password rimbamesh2026. chronite fw 1.17.8, no_hwcrypt=N (HW crypto) — verified.
  • 200 B frames (never fragmented) — isolates the multi-hop / interop question from the fragmentation path (which anyway can't cross to Linux: mac80211 uses the opposite fragment/decrypt order).

Encryption — all four nodes

✓ CCMP-encrypted end to end

chronite: key_mgmt=SAE, pairwise/group cipher=CCMP, wpa_state=COMPLETED; board2 = ESTAB / authenticated / authorized. chronite requires SAE, so board2 being authorized proves board2 did SAE+AMPE+CCMP.

ESPs: peers set to MMWLAN_SAE (not OPEN) at ESTAB (umac_mesh.c:621); g_mesh_sw_crypto=true; SAE+AMPE+CCMP compiled in.

Proof-by-flow (board0+board1): a secured mesh drops unprotected data (umac_datapath.c:702 on the ESPs; CCMP-required mac80211 on chronite). ~15 000 frames flowed both ways across all 3 hops → every frame was CCMP-protected (one plaintext node would black-hole the path).

⭐ A #20 counter-example

The #20 memory states, bench-proven, that a HW-crypto morse node withholds any A4≠TA 4-address mesh forward at the firmware — "HW-crypto mesh multi-hop is impossible on this firmware, on Linux too" (its proof: board1→board0→chronite, board0 TX'd 10 forwards, chronite's raw socket saw 0; direct 40/40).

🐛 This run refutes the "universal / impossible" framing

board0↔chronite here is only reachable as an A4≠TA forward at the board2→chronite hop (A4=SA=board0 ≠ TA=board2), and it delivered at the application layer (ICMP round-trips — stronger than a raw-socket count): board0→chronite 3000/3000, chronite→board0 ~99%. chronite is confirmed HW crypto (no_hwcrypt=N), so this is NOT the "SW-crypto Linux delivers" case #20 already allows.

✓ Reconciled (same night, bench + adversarial-verified) — HALF-refuted

Re-ran the exact 2026-07-11 rig (board1→board0-relay→chronite) + a fully-wired-relay control (board2), measuring app-layer and the original raw-socket method at once:

Topology verified both rigs (board1 LISTEN/BLOCKED, mpath HOP_COUNT 2 via the relay). Adversarial-verified airtight: a forwarded unicast is re-keyed under the next-hop pairwise MTK, and board1 holds a key only with board0 — so a decrypted board1-sourced ICMP at chronite's kernel is cryptographically impossible unless it arrived as the A4≠TA forward; and chronite emitting 223 replies is L3-reception proof independent of the capture method.

  • Pillar A (§3, Linux endpoint) REFUTED — a HW-crypto Linux node can be a multi-hop mesh endpoint. The 07-11 "0" was likely a raw-socket measurement flaw.
  • Pillar B (§1, an ESP FW in HW-crypto as the RECEIVER) NOT re-tested tonight, STILL STANDS — 07-11 measured it at the app layer (0/76 vs a same-rig SW-CCMP control 61/61, crypto-flag-only). Tonight's ESPs ran shipping SW-CCMP, so this pillar was never re-created.

So the memory's "impossible on both stacks / SW-CCMP the only answer for everyone" is over-generalized. Corrected: Linux nodes are not single-hop-only. Practical answer unchanged — the ESPs use SW-CCMP regardless. To fully close: re-run the §1 all-ESP HW-crypto-receiver rig (g_mesh_sw_crypto=false) on the current tree.

board0 stall — TX saturation, not a fault

Diagnosed & fixed by rate, not code

A first bidirectional attempt at 200 ms both ways (~10 pkt/s through board0: 5/s replies to chronite's flood + 5/s own pings) froze board0's esp_ping at ~292 (ok frozen, to=0 → blocked on send, not timing out) while board0 stayed alive. Dropping to 400 ms each way (~5/s through board0) fixed it completely — board0→chronite ran 3000/3000. So the stall was board0's TX queue saturating under the combined reply+ping load; the mesh, crypto, and forwarding were never at fault.

Disposition

No firmware change — the halow working tree is still exactly the defrag-before-decrypt fix (72 ins / 235 del). Temp app scaffolding (MESH_CHRONITE_LINE) git-restored. Radio-silent: 3 ESPs → rimba-hello; chronite wlan1 down + s1g stopped; chronium morse0/wlan1 down.

Gotchas (reusable)

  • To force a Linux node into a multi-hop line, give the ESPs the allowlists (they gate peering + HWMP, so the Linux node becomes a leaf off its single allowed ESP); the Linux node needs no restriction. Set the allowlist before mmwlan_mesh_start.
  • chronite teardown: find the literal -i wlan1 s1g PID and kill it — never killall/pkill -f (self-matches your ssh cmdline; SSH rides wlan0).
  • A sustained inbound ICMP flood can wedge an ESP's esp_ping (blocks on send, no timeout) with no crash — keep the aggregate per-node rate ≲ ~5/s or make the flooded node a responder.
