The three decisive findings

1 · Disjoint histories

git merge-base origin/main esphalow/main → no common ancestor. Ours was a fresh squashed import, so it shares no commit objects with esp-halow. No shared base to rebase onto — replay via cherry-pick or git format-patch … | git am -3.

2 · Version gap 2.10.4 → 2.12

Our diffs were written against 2.10.4; the target is 2.12. A forward-port across SDK versions, not a clean transplant — expect conflicts where 2.12 moved the code our patches touch.

3 · Possible layout mismatch

Ours imported the ESP Component Registry package; esp-halow is the mm-iot-sdk source tree. Layouts may differ, so patches may not apply by path. First step: diff the 68ddbd8d tree vs esphalow/main to map the layout before choosing a strategy.

What "preserve history + PRs" can mean here

  • Commit history: per-commit author/date/message survive cherry-pick / git am, but land as new SHAs on the 2.12 base with conflict-altered diffs — not a SHA-preserving transplant. Be honest about that with the owner.
  • PRs: GitHub PRs can't move across repos. "Preserve PRs" ≈ recreate them (replay each of the 24 as its own branch/PR), or land fewer logical PRs. Numbers + threads won't transfer.

Footguns

  • Submodule gitlink SHA caveat — merge submodule PR(s) so the superproject gitlink resolves on merged main (rebase rewrites SHAs; see .ai/AGENTS.md).
  • Re-run the WHOLE regression suite against 2.12 after repointing — a stack bump is exactly what the suite guards; nothing is done until T0/T1/T2/tp/dscycle re-verify.
  • Standing rules: no commit/push weekday 09:00–16:59, owner-attributed only, code → branch+PR, /review before merge. Bench radio-silent; connect by hostname.
  • fw/driver parity — the 2.10.4→2.12 bump may force a matching Linux driver / MM6108 fw bump on the bench so both sides stay on one version.
  • An esphalow remote was added inside components/halow and esphalow/main fetched during analysis — harmless local-only remote, left for the fresh session.
