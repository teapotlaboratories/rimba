Mesh-gate port S1 — gate RANN TX
written · compiles clean · byte-for-byte identical to a live Linux gate on-air

First stage of the approved 802.11s Mesh-gate port. A gate node periodically floods a RANN (Root Announcement) so every mesh node learns a path back to it — the standards-way replacement for a mesh node's hard-coded default gateway, letting a multi-hop HaLow mesh reach an off-mesh backhaul/internet uplink. S1 is the gate → mesh announcement (TX), follow-Linux and byte-diffable against a real Linux gate.

✓ PASS — gold-standard live A/B (2026-07-21, on-air bench run)

Both gates emitted RANN on ch27 (5560 MHz) at once; chronium (monitor) captured both; the ESP gate's frame is byte-identical to the Linux gate's frame in every field that must match. Verification topology — two gates, one monitor

Bench setup + capture recipe

  • Monitor: chronium — iw dev wlan1 set type monitor + set freq 5560, morse0 up.
  • Linux gate: chronite brought up as a rimba-smesh mesh via wpa_supplicant_s1g (docs/reference/captures/wpa-smesh.conf), then made a gate at runtime with iw dev wlan1 set mesh_param mesh_hwmp_rootmode 4 / mesh_gate_announcements 1 / mesh_hwmp_rann_interval 5000. Readback confirmed rootmode=4 gate_ann=1 rann_int=5000TUs element_ttl=31 — the live gate uses exactly the defaults S1 assumes (iw mesh_param, not iw mesh join).
  • ESP gate: board0 flashed with firmware/test-mesh-gate, which calls mmwlan_mesh_set_root_announcements(true, true, 5000) after mmwlan_mesh_start.
  • Capture script: scratchpad/mesh_rann_cap.py — filters mgmt/action cat=13/act=1/eid=126, dumps hex + decoded fields per SA. Both gates emit RANN as a lone root with zero peers.

✓ Independent static confirmation — 4-lens adversarial workflow (agrees with the bench)

A byte-layout / root-frame-values / runtime-on-air-emission / regression-timer verification returned MATCH on every lens, zero blockers, go=true. Its runtime lens independently proved the broadcast RANN goes out unprotected, triple-safe:

  1. the common_stad is PMF-DISABLED, so the robust-mgmt encrypt block is never entered;
  2. a multicast da bypasses the inner encrypt;
  3. a group-privacy Mesh action is exempt.

So a lone gate with no peer transmits a capturable, in-the-clear RANN — which the bench confirmed.

Footguns

⚠ Interval units — do NOT convert the RANN interval

Linux puts the raw dot11MeshHWMPRannInterval numeric value on the wire (labelled TU but not converted from ms). S1 does the same, so the byte-diff matches as long as both sides use the same numeric value — do NOT apply MESH_MSEC_TO_TU to the RANN interval.
