dscycle deep-sleep tier: reliable + self-contained

The dscycle regression tier proves board2 re-associates on every wake from deep sleep. It had been landing INCONCLUSIVE run-to-run on a flaky wake, and it wasn't self-contained — it left the PPK2 holder to the operator, killed processes with a too-broad match, and silently SKIPped inside test-all. This pass fixes all of it: a commanded C6 wake, a correct cycle count, a self-starting holder, and a precise kill — verified by five consecutive clean runs where there used to be one flaky one.

Bottom line

The tier went from 1 flaky INCONCLUSIVE to 5/5 clean PASS (including a 3/3 and a run that self-started from a fully-powered-off board2). Each deep-sleep cycle is now ~15–23 s wall instead of ~130 s, and the whole tier no longer needs an operator babysitting the PPK2 rail.

1 · The flake — two root causes

Reproducing the baseline (make test-dscycle CYCLES=2, unmodified) gave the usual verdict:

INCONCLUSIVE  333.6s  deepsleep-reconnect
  2/2 reconnects but only 1 long deep-sleep gaps — the board reconnected but did not
  clearly deep-sleep between cycles ... so not a clean deep-sleep-leaf proof.

Cause A — an off-by-one in the cycle count

The DUT logs RECONNECTED in N ms on every association — including the one-off fresh-boot association after flashing, before any deep sleep. The old capture counted all of them and stopped at len(recons) ≥ cycles, so for CYCLES=2 it stopped after fresh-boot + 1 real cycle = 2 reconnects but only 1 deep-sleep gap → the n≥cycles but deep_sleeps<cycles branch fired even when the wake worked.

Cause B — a free-running wake that races the awake window

The C6 fired a LOW pulse every 30 s regardless of board2's state. board2 only arms its ext1 D5-low wake while asleep, and its awake window per cycle is short (boot → connect → hold 4 s → sleep). So a pulse arriving while board2 was awake was wasted, and the board then waited for the next pulse — or fell through to the DUT's 60 s backup timer (~130 s/cycle). Under a less-rested board2 the reconnect itself flaked on top.

2 · The fix — a commanded wake (item #5 enables #1)

The C6 fixture (firmware/test-c6-trigger) no longer picks its behaviour at compile time. It now takes serial commands over its UART0 console and latches a runtime mode (optionally persisted to NVS). The harness holds D5 at Hi-Z, watches board2's USB port vanish (it slept), waits a short settle so the ext1 wake is armed, then commands one pulse — a deterministic wake at exactly the right instant.

The serial protocol (responses prefixed C6|), smoke-tested directly over ttyUSB0:

> ping        -> C6|PONG
> hiz         -> C6|MODE hiz
> pulse 120   -> C6|PULSE ms=120 ... C6|PULSE done -> hiz
> trigger 30  -> C6|MODE trigger period=30s
> bogus       -> C6|ERR unknown: bogus

A /code-review over the diff surfaced 4 low-severity findings (core paths already bench-verified): a leaked log fd in _start_holder (→ with), an unbounded C6 pulse [ms] that could starve the UART RX (→ clamped to 2000 ms; pulse 99999999 → C6|PULSE ms=2000), a stale rimba-hello doc wording (→ test-idle), and a pre-existing IDLE_APP-pre-built dependency (left, noted). Run 5 above is the end-to-end confirmation after those fixes + a C6 reflash.

Design validation: the marker, not the gap, is the proof

Run 3's real gaps are only ~2.5 s — shorter than the old 4 s (and my 3 s) threshold, and close to a bare reset's ~2 s re-enumerate. With the fast commanded wake, a gap-duration heuristic alone cannot tell a real deep-sleep from a reset. So the rework gates on the DUT's DEEP SLEEP log marker (printed only right before esp_deep_sleep_start() with the radio in RESET_N-low) as the primary proof; the gap is corroboration. A pure-gap gate would have false-INCONCLUSIVE'd the very wake we made reliable.

A bug caught mid-verification

Runs 1–2 printed deep-slept 0.0s: a stale /dev/serial/by-id symlink lingers through the vanish transition, so resolve_port briefly returned the old path and the deep-sleep was counted (correctly, via the marker) but with a bogus ~0 gap. Fixed by trusting only a successful serial open as "board2 genuinely back" — run 3 then reported the true ~2.5 s gaps. The PASS was always valid (marker-gated); the fix corrects the reported duration and removes the fragility.

5 · Item-by-item verification

Item 3 — precise kill (the tightened match)

Started a bystander whose command line is literally python3 tools/ppk2_hold.py --bystander but which holds no PPK2 fd. pgrep -af ppk2_hold.py matched both it and the real holder — what the old broad kill would have targeted. _free_ppk2() killed only the real holder (the PPK2 fd owner); the bystander survived.

Items 2 + 4 — self-start from a dark board2

Powered board2 fully OFF with no holder (the post-tp state that used to make the tier SKIP silently), then ran make test-dscycle:

dscycle -- board2 not powered; starting tools/ppk2_hold.py ...
dscycle -- bringing up the AP ...            # board2 enumerated in ~3 s; then PASS 2/2

6 · Files changed

  • firmware/test-c6-trigger/main/main.c — serial-latch firmware (UART0 command loop + NVS persistence); compile-time MODE is now just the power-on default, so tp's free-run is unchanged.
  • tools/regtest/dscycle.py — commanded-wake _capture, correct cycle counting, _ensure_holder / _start_holder, precise _free_ppk2 (fd + pidfile), progress prints.
  • tools/ppk2_hold.py — writes /tmp/ppk2_hold.pid (cleaned up on exit/signal).
  • Makefile, firmware/test-c6-trigger/README.md — docs for the self-start + serial protocol.

Not an on-air change: dscycle is a structural USB/serial gate; the C6 wake is a D5 GPIO pulse, not RF, so no transmitted frame changed and the on-air gold-standard doesn't apply. Harness unit tests (11) still pass; bench left radio-silent (all ESPs on rimba-hello, holder powering board2).
