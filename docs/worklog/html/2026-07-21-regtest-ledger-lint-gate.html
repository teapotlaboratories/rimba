Make the regtest suite unattended: flake ledger + lint gate

Two slices of the "Loop-back: harden" umbrella, driven by one requirement: the harness must run fully automatically — nobody, not even the agent, should have to monitor a run. For that to hold you have to launch it, walk away, and trust the durable record afterwards. Two gaps stood in the way — a latest-wins baseline that erased flakes, and a class of harness bug that only surfaced mid-run on the bench. Both closed.

Bottom line

Every result is now appended to a run-history ledger (never overwritten), so a flake a retry masked is still recorded and make test-flakes surfaces it. And a pyflakes lint gate runs off-bench before every tier, so an undefined-name bug fails in ~0.1 s instead of 40 min into a run. Turning the gate on immediately caught 6 pre-existing findings.

1 · Why the suite couldn't be left alone

Latest-wins erases flakes

Each tier writes build/regtest/<tier>-latest.json and a retry overwrites it. So a passing retry silently deletes the failed run — a flake leaves no trace. Earlier today mesh-linux FAILed on a chronite transient and PASSed on retry; with latest-wins alone, that flake would have vanished. If you're not watching the live run, you never learn it flapped.

py_compile misses the bug that shows up on the bench

A bench-out-of-scope edit once passed py_compile and only failed on the bench, 10/12 FAIL, deep into a run. py_compile compiles to bytecode — it does not resolve names, so an undefined name sails through. pyflakes catches it in ~0.1 s. Without an automatic gate you find out mid-run — exactly when nobody's watching.

2 · The flake ledger

common.Reporter.add() already persists crash-safely after every result — the one chokepoint every tier flows through. It now also appends one JSONL line per result to history.jsonl, append-only, grouped by a per-run id:

make test-flakes prints the ledger; the HTML report carries the same "Flake ledger" section. A dry run records nothing (a new persist flag also closes a latent bug where the per-add() crash-safe write clobbered the real baseline on DRY_RUN=1).

3 · The lint gate (and the 6 bugs it caught)

run.py runs pyflakes over the harness at the top of every run-producing command, and make test-all lints first — a real finding aborts before any hardware is touched, no bypass. A missing pyflakes only warns (never blocks the bench on a missing dev tool). Turning it on flagged 6 findings the harness had been carrying — including the two # noqa-suppressed "reshape later" patterns a prior batch deferred (pyflakes ignores # noqa) and a redefinition of unused 'ap_teardown':

filefinding → fix
t1_smoke.pyunused INCONCLUSIVE import → removed
t2_onair.pyunused capture_serial import → removed; two placeholder-less f-strings → plain strings
tp_power.pyunused REPO_ROOT import → removed; import ppk2_api # noqa probe → importlib.util.find_spec; ap_teardown None-then-def → named helper assigned to the var

4 · Verification

All host-side plumbing — no radio — so it's unit-testable off-bench, which is itself in the spirit of "no monitoring."

  • 26 unit tests pass (make test-unit, now under the IDF python so pyflakes is present): flake detection (flip=flaky, always-pass≠flaky, always-bad=broken, INCONCLUSIVE=bad, SKIP/XFAIL=good), append-is-additive, torn-final-line tolerance, the real Reporter.add()→ledger hook (temp dirs, never touches build/regtest), and a dry-run-records-nothing test; pyflakes catches an undefined name + unused import, passes clean code, and the real harness is clean.
  • Dogfood: make test-lint → clean, 19 harness files after the 6 fixes.
  • Fire-and-forget bench demo: launched 2× make test-t1 board2 detached, wrote these docs without watching it, and read the ledger afterward — the workflow the whole change exists to enable.

The fire-and-forget workflow

make test-all BOARD_NAME=board2 AP=esp CYCLES=3 >/tmp/run.log 2>&1 &
# … later, nothing watched in between …
make test-flakes   # what flipped verdict run-to-run (durable history)
make test-report   # the HTML, incl. the flake section

5 · Numbers that drift (trend + gitlink diff)

The flake ledger catches pass↔fail flips; it does not catch a number that drifts while the test still passes — the tp No-PS current creeping up (the fw-1.17.9 ~2× power-save regression was exactly this), a dscycle reconnect getting slower, throughput sagging. That's the biggest regression class the suite missed. tp/dscycle already record rich numeric meta (median_ma, latencies_ms, …) — the ledger just threw it away.

The command to run after a components/halow bump

make test-trend                          # every metric over recent runs (sparkline + %)
make test-trend DIFF="7d7f76ad 88606f94" # each metric A→B across two halow SDK gitlinks
#   tp:no-ps (No-PS)   median_ma   A=63  B=126   +100%   <-- >20% MOVE

ledger.append_result now persists a bounded numeric meta (scalars, short strings, numeric lists ≤32) per history line; run.py trend shows each metric over the last N runs (a numeric list trends as its mean), and --diff compares two halow gitlinks (SHA prefixes) and flags a >20% move — turning "still passes" into "this number moved N%". The RC / A-MPDU numbers (the survey's next targets) need a public mmwlan_* accessor first (only the internal umac_rc_get_rc_stats exists) — deferred as a submodule follow-on. 7 more unit tests (36 total), incl. the canonical 2× No-PS-mA diff across two SDK gitlinks, + a real-data 2× dscycle demo.
