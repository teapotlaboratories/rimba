Mesh-gate proxy-ARP: reliable mesh↔AP-client resolution
stop depending on lossy No-Ack broadcast ARP · gate app only · uncommitted

The L2 bridge already carries unicast reliably — but ARP resolution crossed it as a broadcast, and mesh broadcasts are No-Ack (lossy). So a mesh node reaching an AP client was hit-or-miss. The gate now keeps both sides' ARP caches populated over reliable unicast, so neither side ever needs to broadcast-ARP across the bridge. Gate app only (rimba-halow-mesh-ap) — no morselib change.

1 · The problem (from the B2 investigation)

The L2 bridge carries unicast reliably, but resolution crossed the bridge as a broadcast ARP — and mesh broadcasts are No-Ack (lossy). So a mesh node reaching an AP client needed its ARP request to reach the far host and the reply to come back, both over lossy multicast — hit-or-miss.

It "worked" only via the client's periodic gratuitous ARP (GARP_TMR_INTERVAL=60 s + TRUST_IP_MAC) — i.e. up to 60 s of latency, and never for a silent host.

2 · What was implemented — two layers, both in the gate app

2.1 · IP↔MAC snoop table

The gate learns each host's IP↔MAC from every frame it already handles — from ARP and IPv4, so it learns a host from ANY traffic (e.g. a ping), not just a once-a-minute gratuitous ARP. Learned in:

  • gw_ap_rx_cb — AP side
  • gw_mesh_rx_cb — mesh group frames
  • gate_ae_rx_cb / S5b — mesh nodes, from the reliable AE frames

Each entry is tagged AP-side or mesh-side.

2.2 · Reactive proxy-ARP

When the gate receives an ARP REQUEST for a host it knows on the other side, it answers directly with that host's REAL MAC (not the gate's) over a reliable link — a unicast on the AP, a proxied unicast (mmwlan_mesh_tx_proxied) on the mesh — and skips the lossy bridge for that frame. Answering with the real MAC keeps the L2-bridge datapath (S5b/S5c) carrying the actual traffic; the gate only short-circuits the resolution.

⚠ Reactive alone is only a partial win

Reactive proxy-ARP still needs the requester's broadcast ARP to reach the gate, and is often masked by the passive gratuitous-ARP learning already happening. On its own it doesn't guarantee reliability — hence layer 2.3.

2.3 · Proactive push — the reliable fix

A 3 s task (gw_arp_announce_task) teaches every learned host about every learned host on the other side over reliable unicast (AP downlink / proxied mesh unicast, both acked). So neither side ever needs to broadcast-ARP across the bridge — resolution stops depending on lossy multicast. With TRUST_IP_MAC on, an unsolicited ARP reply populates the receiver's cache.

4 · Notes / follow-ups

  • The reactive path is masked by passive learning; the proactive push is what provides the reliability guarantee.
  • Snoop table is 24 entries, plain overwrite when full — fine for a small mesh; size it for larger deployments.
  • The push is O(AP × mesh) per 3 s — trivial at bench scale; for a large mesh, rate-limit / only push changed entries.

⚠ Single-subnet assumption

Proxy-ARP + the L2 bridge assume a SINGLE subnet (AP clients + mesh nodes on 10.9.9.x). The gate app still runs the AP on 192.168.12.1 with a DHCP server (two subnets); "retire MESH_GATE_IP / single subnet" makes proxy-ARP the zero-config default rather than needing a static-IP client. Tested here with the STA static on 10.9.9.50.
