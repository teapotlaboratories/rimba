Mesh-gate S5c: AP → mesh ingress
the gate now bridges BOTH directions · verified end-to-end

S5c — the AP→mesh leg of the gate 802.11s L2 bridge — is written, compiles, and is on-air verified (PASS, 3-node mesh+AP+STA). When an AP client sends a unicast frame to a mesh node, the gate proxies it into the mesh as an AE_A5_A6 frame and the mesh node learns the client's MPP. Paired with S5b (mesh→AP), the gate now bridges both directions — the core L2-bridge datapath is complete.

Goal

The AP→mesh direction of the gate L2 bridge, symmetric to S5b's mesh→AP: when an AP client C sends a unicast frame to a mesh node D, the gate proxies it INTO the mesh as an AE_A5_A6 frame (eaddr2=C, eaddr1=D), so D receives the client's traffic and learns mpp(C → gate) for the return path. This mirrors the Linux gate, which xmits the AP client's frame into the mesh with address extension (AE).

The bidirectional bridge

Three ESP32 nodes: a STA associated to the gate's SoftAP, the GATE running mesh (primary) + AP (secondary) on one MM6108, and a mesh destination node. S5c carries the AP client's frame rightward into the mesh; S5b (already landed) carries the mesh node's reply leftward back to the AP client. The gate is the single bridge point between the two L2 domains.

What was implemented — S5c

morselib — new primitive mmwlan_mesh_tx_proxied

In components/halow (uncommitted, +~85 lines): mmwlan_mesh_tx_proxied(final_dst, src, payload, len) injects a proxied AE_A5_A6 frame (eaddr1=final_dst, eaddr2=src, mesh SA = us), resolving where to send by walking the same reachability ladder the datapath already uses. It reuses the S3 umac_mesh_build_forward AE path and the next-hop keying of umac_mesh_forward_data.

Gate app — gw_ap_rx_cb ingress

In rimba-halow-mesh-ap (tracked, modified): gw_ap_rx_cb now inspects the AP RX Ethernet frame. A unicast frame whose dst is not the gate (neither g_ap_bssid nor g_mesh_mac) is re-encapsulated and proxied into the mesh; frames addressed to the gate (DHCP / ping) and broadcast fall through to the existing L3 path. g_ap_bssid is stashed from mmwlan_ap_get_bssid.

Combined S1–S5c morselib diff: 740 insertions across umac_mesh.c / .h / umac_datapath.c. Compile-verified.

✓ End-to-end result

The STA's frame (L2 dst = board1) was delivered by the AP to the gate's gw_ap_rx_cb — resolving a key unknown: the AP forwards a STA's non-BSSID unicast to the host — then proxied into the mesh (eaddr2 = the STA, eaddr1 = board1, 100 B SNAP+ICMP intact), and board1 learned mpp(STA → gate). The gate bridged an AP client's frame INTO the mesh, and the mesh node can now reach the client for return traffic.

Both directions verified

S5b (mesh AE → AP client) + S5c (AP client → mesh) + MPP learning tie the return path together. The core L2-bridge datapath is complete. Radio-silent cleanup done: board0/1/2 → rimba-hello; chronite + chronium wlan1 down.

Not done — L2-bridge finishing touches

  • A true bidirectional round-trip (STA ↔ mesh node ping through the bridge, both ways). The datapath is all present (S5b+S5c+MPP); it needs the mesh node to hold the STA's target IP and reply (its reply uses mpp(STA→gate) → mmwlan_mesh_tx_proxied / send_to_gates → the gate → S5b to the STA). The current ICMP is to 192.168.12.99, which board1 doesn't own, so it drops it — a real round-trip needs the single-subnet IP plan below.
  • Single subnet + retire MESH_GATE_IP (rimba-halow-mesh/app_main.c): put AP clients + mesh nodes on one subnet so reachability is by MAC (via MPP), no L3 gateway. Replace the L3 ip_forward with the L2 proxy as the primary path (they currently coexist).
  • Proxy-ARP / broadcast bridging: a real ARP-driven flow (vs the static ARP used here) needs the gate to bridge broadcast/multicast AP↔mesh — the multicast AE (AE_A4) path, which was out of S3 scope. That's the last datapath piece for a zero-config L2 bridge.
  • S6: live-Linux interop.

Footguns

🐛 Ownership on the proxy path — don't double-free

gw_ap_rx_cb copies the frame + mmpkt_releases it on the proxy path (it owns it); the local path instead hands it to gw_rx_deliver, which releases. Mixing the two frees the same buffer twice.

  • The AP does deliver a STA's unicast-to-non-BSSID frame to gw_ap_rx_cb — S5c relies on it, confirmed on-air.
  • The STA's AP-MAC is the MM6108 MAC (bc:2a:33:96:b2:33), read from the gate AP client joined log — not the efuse MAC.
  • s_ap2mesh / s_ae2ap static buffers are safe (single mesh-RX task); mmwlan_mesh_tx_proxied copies the payload.
