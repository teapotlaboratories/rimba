Every tier green except dscycle, which is honestly INCONCLUSIVE (flaky C6 wake — not a code regression). All exit codes 0.

The symptom — 3 FAILs, one shape

The three first-run FAILs were exactly the tests whose AP is test-apsta-ap on board0 — ap-sta-ping, twt, multi-twt — all failing at an identical ~52 s with support role 'ap' (board0) did not come up: up-marker 'ap-ready' not seen within 25s. Every mesh / gate / Linux-AP test passed. The AP had clearly booted (chip 0x0306, fw 1.17.8, netif up=1) — it just never emitted the marker.

Root cause — the app_main task stalls, the AP does not

Direct reproduction on board0 (full raw console) pinned it: the AP prints netif up=1 at 3.4 s, then the console goes completely silent — no panic, no reset — and ap-ready, the very next statement, never appears.

  • Reproduced on all three boards incl. fully-wired board2 → not the WAKE/BUSY wiring.
  • Not the rename: the test-apsta-ap code path is byte-identical to the pre-rename regtest-apsta-ap (only the REGTEST→TEST strings changed).
  • Not the close-bench TX cap: removing mmwlan_override_max_tx_power(1) did not change it.

✓ The decisive test — the AP is fully alive

With board2 running the "hung" AP, a real STA (test-apsta-sta on board1) associated (SAE, 3.75 s) and got 15/15 ICMP replies, RESULT=PASS. So the SoftAP is serving perfectly — only the app_main task stops emitting console output after the netif comes up, so the up-marker never reaches the orchestrator. A textbook false-FAIL.

The fix

Emit the ap-ready up-marker before assign_static_ip() — i.e. before the netif-up call that stalls the print stream — instead of after it (firmware/test-apsta-ap/main/app_main.c). The static IP is still pinned microseconds later, and the orchestrator only starts flashing the STA tens of seconds after seeing the marker, so the AP is answering ICMP long before any association.

Open follow-up

⚠ Root-cause the app_main stall (separate firmware issue)

app_main stops producing console output right after esp_netif_action_connected brings the AP netif up (the SoftAP itself keeps running). It did not stall in the 2026-07-17 14:00 run, so a clean rebuild against the currently-modified components/halow surfaced it. Candidates: a main-task stack overflow exposed by the rebuild's layout, or an esp_netif/morselib AP-path change. Start from a submodule-state diff (or bump CONFIG_ESP_MAIN_TASK_STACK_SIZE). The interim marker move keeps the suite honest meanwhile.

Update 2026-07-21 — RESOLVED (a mid-migration artifact)

✓ The stall does not reproduce on the shipped 2.12.3 SDK

Reproduced on-bench on board2 with an instrumented build (a heartbeat + a stack/heap probe right after the netif-up), captured from a fresh boot twice: app_main printed straight through esp_netif_action_connected — ap-ready + netif up=1 at 3.8 s, then a heartbeat every 1 s for 30+ s past the netif-up, no silence, no crash (consistent across both runs).

Both suspects are ruled out: the main-task stack HWM is a healthy 1528 B free (of 3584), and there is no esp_pm/light-sleep on the ESP build (CONFIG_PM_ENABLE unset). Diagnosis: the 2026-07-18 stall was an artifact of the mid-migration components/halow (the "currently-modified" submodule during the 2.10.4→2.12.3 forward-port). The exact intermediate cause can't be pinned — that uncommitted build state is gone — but it is absent on the merged 2.12.3 that ships (7d7f76ad).

Fix: the interim workaround is reverted — the ap-ready marker is back at its natural position (after assign_static_ip, so it truthfully means "up + IP pinned"); diagnostics removed. Verified end-to-end: make test-t2 TEST="ap-sta-ping" → PASS (the orchestrator saw the marker + the STA associated + pinged). Uncommitted; held for a post-17:00 commit.

Bench returned to radio-silent after: all 3 ESPs on test-idle (zero HaLow lines), chronite wlan1 DOWN, board2 re-held on the PPK2 rail.
