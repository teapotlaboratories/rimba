The "??" that looked like dead hardware
S3 MTK desync + plid-IGNR fix — and two corrections

Set out to verify a follow-Linux fix for the S3 relay forward-drop (a pairwise MTK key desync). Two things turned out wrong, and both are worth more than the fix: a build-config mistake masqueraded for a whole bench session as dead radio hardware, and once that was cleared, the plid-IGNR fix did not actually fix S3. The one solid win: every board is healthy, and there's now a firm build rule so this can't recur.

⚑ Correction 1 — "dead MM6108 / degraded silicon / power" was a build bug

Every firmware was built with bare idf.py, which ships CONFIG_HALOW_COUNTRY_CODE="??". morselib (umac_interface.c:266) refuses to bring up the radio when the country is "??" — it returns MMWLAN_CHANNEL_LIST_NOT_SET before it ever talks to the chip. The symptom is indistinguishable from dead hardware (garbage chip id, MAC 00:00:00, version UNAVAILABLE). Building via make … BOARD=proto1-fgh100m (country US) booted every board on the first try. The hardware was fine all along.

⚑ Correction 2 — the plid-IGNR fix does not fix S3

Re-verified on a proper build: the multi-hop forward still MIC-fails 99.3% (sw_ccmp_fail=2061/2075), keys still desynced, and the failure is immediate from ESTAB, not after a re-peer — so the plid re-peer guard targets the wrong mechanism. It stays as a legitimate hardening; S3 stays open with a sharper lead.

The tell: garbage that varied

The chip id printed as 0x8200f5e7, then 0x8200d052, then 0x0000 across reads — it changed. That's the fingerprint of uninitialised memory, not a real register read: mmwlan_get_version was failing and leaving the struct un-populated. The BCF still parsed (mf16858) because that's a host-side blob parse, not a chip transaction. The chip was never being spoken to.

The fix is to build through the Makefile, which layers the board overlay (boards/proto1-fgh100m/sdkconfig.defaults → the MM6108 pins, partition, and country="US") via SDKCONFIG_DEFAULTS:

# wrong — drops the board overlay, ships country "??"
idf.py build

# right — layers boards/proto1-fgh100m/sdkconfig.defaults (country US, pins, partition)
make build APP=rimba-halow-mesh-perf BOARD=proto1-fgh100m

What the build bug contaminated

Every test that ran on a "??" build had a non-functional radio, so its "still fails" result only re-proved the country gate — it established nothing about the real cause:

⚠ Suspect by association

Earlier-session notes about a "warm-reset-wedge / degraded ESP MM6108s" carry the same "??" signature (CHANNEL_LIST_NOT_SET, garbage id, MAC 00:00:00) and may also be this build bug rather than a real hardware wedge — re-check the build method before trusting them.

The plid-IGNR fix — kept as hardening, not the S3 cause

The fix itself follows mac80211 exactly. Its premise: the per-pair MTK is derived once at plink ESTAB and never re-derived, but umac_mesh_handle_peering_action unconditionally learns a re-peering peer's new nonce/plid — so a peer that restarts can leave the two ends on different MTK epochs. The guard mirrors mesh_plink.c:1077-1088: ignore an OPEN/CONFIRM whose plid ≠ the recorded plid (OPN_IGNR/CNF_IGNR), before learning the new plid/nonce.

That is a real, correct improvement — but the S3 forward-drop is not a re-peer desync, so it doesn't move the needle here. The fix stays in the tree as follow-Linux hardening.

S3 is still open — the real lead

On a proper build, the multi-hop forward board0 → board2(relay) → chronite still fails at the relay's host SW-CCMP decrypt:

board2 decrypt OK14 / 2075
board2 MIC fail2061 / 2075

The dual-side CCMP dump shows the same forward frame (A1=board2, A2=board0, A3=chronite) encrypted and decrypted under different keys — the board0↔board2 MTK the two ends use for the forward do not match:

board0 encrypts forward with
6f 1d 9d 9c 48 af d6 49
board2 decrypts with
7e 58 66 27 00 89 00 51

But single-hop board0 → board2 direct works, so that MTK is synced — meaning the forward path selects the wrong key. board0's TX-DBG confirms it: nh_eq_stad=2930 forwarded frames encrypted with the wrong per-link key. The leading hypothesis: the forward encrypts under the destination (chronite) MTK instead of the next-hop (board2) MTK — the S2 key_stad override in umac_datapath, not AMPE derivation and not the plid re-peer.

⚠ Confound to fix before re-diagnosing

The bench boards are too close (−3…−25 dBm), so board0 peers chronite directly and therefore has a chronite key to grab by mistake. A clean re-diagnosis needs a forced true-multi-hop topology where board0 cannot reach chronite directly.

Disposition

  • Hardware: healthy — board0/1/2 all boot (chip 0x0306), board1's swapped module included.
  • Build workflow: ruled — always make … BOARD=proto1-fgh100m; never bare idf.py (.ai/AGENTS.md §Building).
  • plid-IGNR fix: kept as follow-Linux hardening; uncommitted; does not resolve S3.
  • S3 forward-drop: re-opened with a sharper lead — the forward key_stad selection, to be re-examined on a forced multi-hop rig.

Bench radio-silent after (all three ESPs powered off, chronite wlan1 down).
