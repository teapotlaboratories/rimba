ESP mesh-relay Interrupt-WDT crash: code-only root cause
the fatal path is a recovery teardown, not a hardware wall — the relay just arms it far more often

Root-caused from code alone (bench radio-silent, board2 PPK2-unpowered), adversarially verified. Verdict: software bug — not a hard ESP32 capacity wall (high confidence). Under sustained forward load the relay's mesh-forward blocks up to 1000 ms in the umac-core evtloop task, stalling all mesh RX processing until the resulting comm/pager failures pile to 30 and escalate to a full hw_restart. That restart frees and re-allocates the ESP SPI host controller + its interrupts across cores via blocking IPC — and that is what trips the 300 ms interrupt watchdog. The two panic leads (a core in gpio_isr_loop; esp_intr_alloc on an ipc_task) are the two sides of that cross-core stall. Nothing implemented, nothing committed.

✓ The verdict — software, fixable following Linux

An INT_WDT panic (with TASK_WDT_PANIC disabled) fires only when a core keeps interrupts disabled >300 ms. A throughput/worker/memory ceiling produces backpressure and loss — never a 300 ms interrupt-disable. Add the byte-identical originator staying rock-stable and abundant in-code software slack, and the hard-wall thesis is refuted. The relay is a restart-frequency amplifier, not a distinct crash class — the fatal teardown is role-independent (also reachable via a firmware watchdog).

This entry is standalone: the failure, the three-stage chain with anchors, the relay-vs-originator delta, the refuted hypotheses, the software-vs-wall reasoning, the ranked fixes and the bench plan are all here.

1 · The failure and the two panic leads

  • A relay node (board1 → board0(relay) → board2) crash-loops on INT_WDT — 300 ms, dual-core (ESP_INT_WDT_TIMEOUT_MS=300, INT_WDT_CHECK_CPU1=y; ESP_TASK_WDT_PANIC not set, TASK_WDT_TIMEOUT_S=5).
  • A node that only originates on the same firmware is rock-stable; a Pi-5 Linux relay never fails.
  • Panic leads: (A) a CPU in gpio_isr_loop; (B) gpio_isr_register_on_core_static → esp_intr_alloc on an ipc_task.
  • Correlated (not proven causal) with opening the relay's USB-CDC console mid-run (XIAO-S3 DTR/RTS).

2 · The root-cause chain

Stage 1 — the forward blocks the umac-core evtloop task confirmed

◑ Correction (found while implementing FIX-2/S3)

An earlier draft placed this block inside the driver-task bus_lock, starving the SPI worker. That was wrong. umac_datapath_rx_frame (:1933) only queues the RX frame + umac_core_evt_wake (:1945-1947); umac_datapath.c holds no bus_lock. The forward runs later in the separate PRI_HIGH umac-core evtloop task (umac_evtloop.c:109). The block is real and still the arming mechanism — but it stalls the evtloop, not bus_lock. Verdict + fixes unchanged.

The driver task reads RX pages under bus_lock but only queues each frame to the evtloop; the forward (and its blocking wait) happens in the evtloop:

[driver task, under bus_lock]            pageset.c:985  morse_trns_claim(bus_lock)
  morse_pageset_rx_handler → morse_skbq_process_rx → mmdrv_host_process_rx_frame   skbq.c:329
    umac_datapath_rx_frame                 umac_datapath.c:1933
      umac_datapath_rx_queue_frame + umac_core_evt_wake   :1945-1947   ← QUEUES, returns
  morse_trns_release(bus_lock)             pageset.c:1077

[umac-core "evtloop" task, PRI_HIGH — holds NO bus_lock]   umac_evtloop.c:109
  umac_datapath_process_rx → …reorder… → umac_mesh_forward_data   umac_datapath.c:3117, :871, umac_mesh.c:2573
    umac_datapath_tx_mesh_keyed_frame       umac_datapath.c:2694
      wait_for_tx_ready_( calculate_tx_timeout_ms(umacd, true) ) = 1000 ms   mmwlan.h:2821

For a relay, the evtloop reaches the forward, does inline SW-CCMP encrypt, then blocks up to 1000 ms in wait_for_tx_ready — stalling the one task that drains all mesh RX. While it is blocked, the driver task keeps queuing RX the stalled evtloop can't drain → RX/pktmem backlog → pager/comm failures pile up (Stage 2). The precise evtloop-stall → failure-count chain is bench-gated; the code-confirmed fact is that the block is in the evtloop, not bus_lock.

Stage 2 — transient SPI failures escalate to a full restart confirmed

comms_op_check (sdio.c:121) counts failures; at MAX_COMM_FAILURES=30 it masks the IRQ and calls driver_health_demand_check. The PRI-LOW health task's 1-retry morse_cmd_health_check fails → morse_reset_chip (driver_health.c:41) → mmdrv_host_hw_restart_required.

Stage 3 — the restart re-allocates the SPI host + interrupts across cores path confirmed · duration inferred

hw_restart_evt_handler (umac_mmdrv_shim.c:68) runs mmdrv_deinit() then mmdrv_init() → spi_bus_free + spi_bus_initialize → esp_intr_free/esp_intr_alloc. Because every morselib task is unpinned (xTaskCreate = tskNO_AFFINITY, mmosal_shim_freertos_esp32.c:220), the free/alloc routes to the owner core via esp_ipc_call_blocking → the ipc_task = backtrace B (the only post-boot intr alloc/free site). It holds the global intr-alloc spinlock (intr_alloc.c:121) — shared with ISR-context shared-vector dispatch — so the other core spins on it with interrupts disabled = backtrace A. Either stuck core → INT_WDT.

3 · Relay vs originator — frequency, not susceptibility

The forward branch (umac_datapath.c:871, gated mesh_da != us) is reachable only from the umac-core evtloop's RX processing; an originator never reaches it. The originator runs the identical TX from the app/lwIP task — its identical 1000 ms wait blocks only its own caller, never the evtloop that drains mesh RX. So the relay's forward stalls its own RX pipeline and arms the restart far more often.

But the fatal teardown is role-independent — it runs on any restart trigger, including the MORSE_INT_HW_STOP_NOTIFICATION firmware watchdog (hw.c:75) and, plausibly, the console-open flash stall. An originator is not immune; it merely triggers the restart far less often. The relay amplifies restart frequency; the crash mechanism is the same for everyone.

4 · What was refuted

HypothesisWhy it fails
Level-triggered SPI IRQ spins gpio_isr_loop during teardownThe pin is masked GPIO_INTR_DISABLE three times before spi_bus_free (sdio.c:129, driver.c:515, sdio.c:839) and RESET_N is lowered first (mmhal_wlan.c:245) — the IRQ source is de-asserted.
Pure level-IRQ livelock, no restartISR masks before the loop clears status; loop is IRAM, bounded, single-snapshot; sole re-enable is in the worker task (1:1 handshake). Also role-agnostic.
Priority inversion on bus_lockbus_lock is a mutex with priority inheritance (mmosal_shim_freertos_esp32.c:325); the self-throttle isn't inversion.
Hard ESP32 capacity wallA throughput ceiling gives backpressure/loss, never a 300 ms interrupt-disable panic; identical-HW originator is stable.

5 · Ranked follow-Linux fixes

◑ Follow-Linux caveat — verify on chronite first

morse_driver / net/mac80211 are not checked out on this disk (only a .git/modules stub; the live tree is on chronite ~/halow). The Linux "how it's done" claims below are convention and must be grep-verified on chronite before implementing FIX-1/2/3.

True root-cause fixes

  • FIX-1 (primary — removes the panic for all triggers): split "chip reset" from "SPI-host teardown." Make hw_restart toggle RESET_N + reload FW without spi_bus_free/spi_bus_initialize or gpio_isr_handler_remove/add — keep the ESP SPI peripheral + esp_intr allocation persistent (Linux never re-probes the host controller). Removes backtrace B and the whole cross-core window. Targets: umac_mmdrv_shim.c:87-88, sdio.c:836/769, mmhal_wlan.c:241/233. Risk: medium.
  • FIX-2 (removes the relay trigger; also the S3/B6 relay-aggregation win): non-blocking forward off the evtloop's blocking wait. Minimal = flip the forward wait to calculate_tx_timeout_ms(…, false) (drop-on-full). Proper = queue the forward to a work item instead of running it inline in the evtloop RX processing (mirrors ieee80211_rx_h_mesh_fwding + a tx workqueue). Risk: low → medium. Do this together with S3.
  • FIX-3 (proportionate recovery): retry/backoff transient CMD52/53 errors in comms_op_check (sdio.c:121); reserve the full restart for a confirmed FW watchdog, not a raw count of 30 transient failures.

Defense-in-depth hardening

  • FIX-4: mask before free in mmhal_wlan_deinit (gpio_set_intr_type(DISABLE) before spi_bus_free). Closes the latent level-line window. Risk ≈ zero.
  • FIX-5: pin the driver tasks to one core (xTaskCreatePinnedToCore) so esp_intr_free frees locally instead of via cross-core IPC. Risk: medium.
  • FIX-6: IRAM-safe GPIO ISR (GPIO_CTRL_FUNC_IN_IRAM=y + ESP_INTR_FLAG_IRAM + IRAM_ATTR) so masking can't stall on a flash-cache-disable window — addresses the console-open correlation. Risk: low.

6 · Bench repro + validation (relay has no console)

Restore: repower board2 (tools/ppk2_hold.py), confirm fw 1.17.8 everywhere (dmesg fw SIZE 480664), bring up board1 → board0(relay) → board2. Instrument without a live UART:

  • RTC-noinit counters (survive the panic): hw_restart_count, peak consecutive_comm_failures, forward-wait timeouts, bus_lock max-hold-µs. Proves Stage 1–2; originator stays ~0.
  • GPIO scope strobes around hw_restart_evt_handler + esp_intr_free/alloc — measures the Stage-3 interrupts-disabled duration (the one inferred link).
  • esp_core_dump to flash — both cores' PC + interrupt state offline; resolves whether backtrace A is a spinlock spin vs. an innocent CHECK_CPU1 snapshot.

A/B: flip FIX-2 minimal alone → expect hw_restart_count → 0 under the same load; FIX-1 alone (force a restart via a test hook) → expect no INT_WDT. After every test: radio-silent — rimba-hello to all ESPs + ip link set wlan1 down on any Linux node.

⬡ Method

Root-caused code-only via a 3-phase agent workflow (6 causal lenses → 3 adversarial verifications incl. the hard-wall counter-thesis → synthesis), then re-verified by hand. The task context of the blocking forward was corrected twice — off the SPI worker onto the RX drain, then (during the FIX-2/S3 implementation pass) to the umac-core evtloop task, not the driver-task bus_lock (Stage 1 correction). The relay claim is frequency-not-susceptibility.
