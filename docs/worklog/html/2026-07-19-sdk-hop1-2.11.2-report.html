✔ Bottom line

The upstream refactor swapped a single global datapath ops table for a per-VIF ops model. We deleted our custom mesh-gateway machinery and re-expressed mesh + AP concurrency natively on it. Six fixes later — none a re-arch logic bug — STA, IBSS, TWT, SW-CCMP mesh, A-MPDU, mesh peering/relay, AP↔STA, and the concurrent mesh-gate are all green. Uncommitted; tagging 2.11.2-esp32-1 is the owner's call.

What "the hop" actually is — three layers

The whole change sits on one commit over the pristine 2.10.4 base (3c8ded0d):

1 · The architectural shift: global ops → per-VIF ops

The load-bearing change. On 2.10.4 the datapath kept one ops table (data->ops) and one active data->vif_id. On 2.11.2 each interface owns its ops table in a two-slot structure; TX frames carry a stamped tx_metadata->vif_id, RX frames are demuxed by the firmware's rx vif.

Before · 2.10.4 
/* one table, one vif */
struct umac_datapath_data {
  const struct umac_datapath_ops *ops;
  uint16_t vif_id;
};

/* TX: 5-arg, vif passed in */
umac_datapath_tx_frame(
  umacd, pkt, enc, ra, vif_id);
After · 2.11.2 
/* two fixed slots, each with ops */
struct umac_interface_data {
  umac_interface_vif_data data_sta;
  umac_interface_vif_data data_ap;
};

/* TX: 4-arg, vif via metadata */
umac_datapath_tx_frame(
  umacd, pkt, enc, ra);

2 · Re-architecting the mesh-gate onto per-VIF

Our biggest feature — the all-ESP mesh-gate (mesh primary + a concurrent SoftAP on one MM6108, L3-routing between them) — was built on the old global model with a bespoke datapath_ops_gateway that served both stads. That model is gone. The owner's call was "re-arch fully clean — delete the fallbacks, trust the FW vif tagging."

Deleted (our gateway machinery) 
- datapath_ops_gateway
- gw_*  (helpers)
- configure_gateway_mode()
- data->ap_vif_id
Created (native per-VIF tables) 
+ umac_datapath_mesh.c  (250 L)
+   umac_datapath_ops_mesh → VIF_STA
+ umac_datapath_ibss.c  (191 L)

3 · Firmware: uniformly 1.17.8 (no version change)

⚠ Correction to record

An earlier draft framed this as a "1.17.6 → 1.17.8 firmware-parity bump." That was wrong. The bench — all 3 ESP nodes and all 4 Linux nodes — runs the identical chip firmware 1.17.8 (mm6108.bin, 480664 B, md5 cfe56db2), and has for the whole HaLow effort.

The reason it's identical is the build system. cmake/mm-fw-gen/ treats vendor/morse-firmware as the single source of truth and regenerates the ESP-loadable Morse-TLV .mbin from that ELF at build time (convert-bin-to-mbin.py). Per the generator's own header:

/* "Nothing firmware-related is committed under components/; the firmware
    version is whatever vendor/morse-firmware is pinned at." */

So the mm6108.mbin committed inside the morselib submodule is never used — a dormant artifact of the vendored SDK package. I bumped it (399468 → 399796 B) to 2.11.2's blob for tree-fidelity, but it has zero build effect: the ESP loads the same 1.17.8 binary the Linux nodes do, regenerated from vendor/morse-firmware, before and after. There was no firmware version to reconcile; fix #2 is a 2.11.2 mesh-vif quirk on the unchanged fw.

4 · The fixes

Regression testing surfaced five real breaks. The through-line: the re-arch's structure was sound — the bugs were latent assumptions in our glue, one-per-VIF-model corners the global model had masked, and one mesh-vif/beacon-config quirk on the (unchanged) 1.17.8 firmware that the 2.11.2 setup surfaced.

1

get_tx_seq — a key type where a key id was expected

Upstream moved the key_type → key_id resolution out of connection_keys; our resolution kept the enum, so a key type flowed into a slot that wanted a key id — corrupting the mesh SW-CCMP packet number.

/* umac_keys.h / .c + umac_datapath.c */
- uint64_t umac_keys_get_tx_seq(stad, enum umac_key_type key_type);
+ uint64_t umac_keys_get_tx_seq(stad, uint8_t key_id);
- seq = umac_keys_get_tx_seq(stad, key_type);
+ seq = umac_keys_get_tx_seq(stad, (uint8_t)key_id);
2

Mesh vif wouldn't start — fw 1.17.8 rejects a standalone beacon config

The mesh vif failed at mmwlan_mesh_start: the firmware returned fw_status=14 to a standalone BSS_BEACON_CONFIG. The firmware is 1.17.8 the whole time (§3) — what changed is the 2.11.2 mesh-vif bring-up, which trips a rejection the 2.10.4 path didn't. The Linux 1.17.8 driver confirms the fix's shape: it doesn't send config_beacon_timer on the initial mesh start (MESH_CONFIG(enable_beaconing) arms beaconing). Following that reference, we made it non-fatal.

  ret = mmdrv_config_beacon_timer(vif_id, true);
- if (ret != 0) { goto fail; }
+ if (ret != 0) {
+   /* fw 1.17.8 rejects standalone BSS_BEACON_CONFIG (fw_status 14); Linux doesn't send it on
+    * the initial mesh start — MESH_CONFIG(enable_beaconing) arms beaconing. Non-fatal. */
+   MMLOG_ERR("MESH: config_beacon_timer fw_status=%d (non-fatal on 1.17.8)\n", ret);
+ }

verified mesh vif up + beaconing → mesh-peering, mesh-relay, ampdu-cap PASS.

3

AP SoftAP: 0 ICMP replies — egress left the VIF UNSPECIFIED

A STA associated and completed the 4-way, but every ICMP reply died at mmwlan_tx_pkt: Unable to infer VIF ID. Root cause is in our ESP glue (mmhalow.c): halow_transmit left .vif = 0 = UNSPECIFIED. Under the old global model that resolved via the single data->vif_id; the per-VIF inference bails when both a STA vif and an AP vif are valid — and mwlan always boots a STA vif, so a plain AP app is ambiguous.

/* components/halow/mmhalow.c — halow_transmit + mmhalow.h field */
+ mmhalow_netif_driver_t *driver = (mmhalow_netif_driver_t *)h;
  struct mmwlan_tx_metadata metadata = {
    .tid = 0,
+   .vif = driver->tx_vif,     /* was implicitly UNSPECIFIED */
  };
/* mmhalow_wifi_start (AP path): */
+ morse_drv->tx_vif = MMWLAN_VIF_AP;

verified ap-sta-ping → 15/15 ICMP (was 0/15).

4

Mesh-gate: a new STA can't associate — the reassign LUT only ran on INVALID

With mesh + AP concurrent, the firmware tags a client's broadcast PROBE_REQ with the mesh vif (a valid id, not INVALID). The reassignment LUT (PROBE_REQ → AP) only ran on an INVALID fw vif, so the probe went to the mesh ops and the AP never answered. Instrumentation showed PROBE_REQ vif=0; after the fix, probe/auth/assoc/action all vif=1 (AP).

/* umac_datapath.c — apply the LUT authoritatively (no-op unless the target vif exists) */
- if (vif_id == MMDRV_VIF_ID_INVALID) {
-   reassign(umacd, metadata, fvts, lut, N);  /* only on INVALID */
- }
+ bool lut_managed = reassign(umacd, metadata, fvts, lut, N);  /* always; now returns bool */
+ vif_id = umac_interface_get_vif_id_from_rx_metadata(metadata);
+ if (vif_id == MMDRV_VIF_ID_INVALID && !lut_managed)
+   MMLOG_WRN("RX VIF unknown ...");   /* warn lifted out of reassign */

verified STA associates to the concurrent gate AP · mesh-peering still PASS (no mesh regression).

5

Mesh-gate: forwarding STA → mesh was dead — the gate's mesh netif egressed UNSPECIFIED

Association fixed, but the STA got 0/15 replies from the far mesh node. The gate's mesh netif is mmhalow's default, and the gate calls mmwlan_mesh_start directly (not mmhalow_wifi_start), so its tx_vif stayed UNSPECIFIED — ambiguous once the concurrent AP exists, so the mesh-bound forward was dropped. A small new API lets the gate tag its mesh netif STA.

/* components/halow/mmhalow.{h,c} — new API */
+ void mmhalow_set_tx_vif(enum mmwlan_vif vif);
/* firmware/test-mesh-ap-gate/main/app_main.c */
  g_mesh_netif = mmhalow_get_netif();
+ mmhalow_set_tx_vif(MMWLAN_VIF_STA);   /* mesh = STA host-slot; morselib resolves STA→MESH */

verified mesh-ap PASS (full STA→far-node ping) · mesh-ap-multi-twt PASS.

6

twt-assoc regression — the assoc-time TWT seed was compiled out by a #if guard

⚠ Correction to the record

The first draft listed twt-assoc's INCONCLUSIVE as a "pre-existing accessor limitation, not a regression." That was wrong — it PASSED on 2.10.4 (2026-07-17) and only INCO'd on 2.11.2, a genuine Hop-1 regression.

The hostap 2.11.2 forward-port tightened the assoc-time TWT-seed guard in wpa_supplicant.c (two sites). The ESP build uses the mmwlan driver and defines MM_IOT, not CONFIG_DRIVER_NL80211_MORSE — so the whole TWT-config block compiled out: the .twt_conf op never fired → no agreement seeded → the assoc request carried no TWT IE → INCONCLUSIVE. It compiles fine either way (why compile-verify missed it — the flagged 268-file hostap bench-risk).

- #if defined(CONFIG_S1G_TWT) && defined(CONFIG_DRIVER_NL80211_MORSE)
+ #if defined(CONFIG_S1G_TWT) && (defined(CONFIG_DRIVER_NL80211_MORSE) || defined(MM_IOT))

verified serial probes show the chain light up (.twt_conf fires → agreement → assoc-req IE → AP grant → INSTALLED flow 0) + harness twt-assoc PASS → full T2 12/0.
