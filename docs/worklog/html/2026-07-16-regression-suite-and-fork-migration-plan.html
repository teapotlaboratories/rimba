Regression suite (T0–T2) + fork-migration plan

Two pieces of work, in order. Phase 1: build the regression suite the backlog has wanted since the start, and get it green on today's code. Phase 2: plan — not execute — the migration of mm-esp32-halow into a real fork of MorseMicro/esp-halow, history preserved. The order is the point: a green baseline before the migration is the only way to tell a migration break from a pre-existing one.

Starting state — no harness existed

There was no test target. tools/ held only ppk2_hold.py, esp_usb_power.py, mesh_ccmp_verify.py, mesh_grab_fwd.py; the Makefile had build/flash/monitor/… and nothing else. 17 apps under firmware/, 2 boards. A single make build is ~1m17s — a full matrix is 30–80 min, long enough that the runner had to be resumable and unattended, not a foreground loop.

The three tiers — and what each does not prove

Each tier is explicit about its ceiling. A test that overclaims is worse than no test, because it gets trusted.

TierNeedsProvesDoes NOT prove
T0 buildnothingevery app × board compiles via make, with a real country code baked inanything at runtime — a T0-green tree can still have a dead radio
T1 smokeone boardflash + boot + the radio really comes up (real chip id / fw / MAC / runtime country)any on-air feature — a T1 board has no peer
T2 on-aira real rigthe milestone claims that matter (assoc, IBSS, TWT, mesh peering/relay, mesh+AP, SW-CCMP)throughput / power numbers — those are benchmarks, not gates

T0's first run found a real, pre-existing break

🐛 BOARD=proto1 cannot build any app

boards/proto1/sdkconfig.defaults:33 asks for bcf_mf16858.mbin. The pinned vendor/morse-firmware (fd41e1c, 1.17.8) ships no mf16858 BCF at all — every proto1 build fails at CMake configure with "BCF ELF 'bcf_mf16858.bin' not found". The BCF worked before the 2026-06-24-firmware-vendor-sourcing change moved firmware to the submodule, which silently dropped it. The whole bench is proto1-fgh100m, so nobody had noticed.

This validates the tier design on its first run: T0 needs no hardware and found a broken board in ~25 seconds. It is not a regression this session introduced — it predates it. Rather than paper over it (a BCF is per-module RF calibration; the dev plan warns against flashing the fgh100m BCF onto mf16858 hardware) or hide it (dropping proto1 from the matrix), the manifest marks proto1 known-broken with the evidence and T0 reports those builds as XFAIL — counted, printed, non-gating, and flips to a loud XPASS the moment someone restores the BCF.

T1 + T2 on real silicon

board2 (the only fully-wired ESP, needed for load/relay/PS) was not enumerated this session — its ppk2_hold.py was down — so bring-up tiers ran on board0, valid because they exercise only light-load bring-up where board0's unwired WAKE/BUSY do not matter.

✓ T2 — 9 on-air feature tests automated + passing (8 all-ESP + 1 ESP↔Linux)

A multi-role orchestrator flashes each test's roles (support first, verified via a console up-marker), scrapes the reporter's REGTEST|RESULT, and returns every board to rimba-hello. Every ESP role app is regtest-*; a Linux node is a support role driven over ssh. The 8 all-ESP tests:

testproof
swccmpRFC-3610 CCM KAT, rc=0 (no radio, deterministic)

mesh-peeringSAE+AMPE plink → ESTAB, ESP↔ESP
mesh-relayforced line, sole peer = relay (board2), 14/15 forwarded
mesh-apSTA→gate→mesh→far-node, ttl=63 (one gate hop)

Plus the gold-standard 9th test, mesh-linux: the ESP peers (SAE+AMPE) with a real Linux node (chronite) and pings it — proving interop with genuine mac80211 + morse_driver, not just another ESP running the same morselib. The Linux side is fully orchestrated over ssh (linux_peer.py pushes the mesh config, starts wpa_supplicant_s1g, and tears the node back down). Verified: ESTAB peer = chronite 3c:22:7f:37:51:38 + 13/15 replies.

The verdict lives in the firmware (only the device sees plink/peer/crypto state — a host scraper of log prose is the "endpoint logs are not proof" anti-pattern). Three small mmwlan* accessors were added to morselib so reporters can self-verify a structural fact that lives in an internal struct (TWT agreement state, A-MPDU cap, IBSS peer count).

Design decision: the verdict lives in the firmware

Each regtest-* app runs the feature and prints one fixed REGTEST|RESULT|PASS|… line; the harness only transports it. Only the device can see association / plink / peer-table / crypto state, and "endpoint logs + a working ping are not proof" is already a project rule. A host scraper asserting from log prose would reintroduce exactly that anti-pattern.

Design decision: assert the structural fact, not the RF number

The bench's own docs are emphatic that RF numbers here are noisy (RX overload at close range; a power-marginal node; the mesh ceiling is "the link, not the code"), and the recorded milestone numbers already include single-packet misses (IBSS "4/5", AP "7/8 then 5/5") — a hard equality assert would have flaked on the very run that produced the number. So T2 asserts "peered + forwarded with ccmp_fail==0" and reports throughput, never gates on it. Noisy expectations are tagged in the manifest with the weaker assertion spelled out.

One morselib change was required (and is upstream-faithful)

morselib's build runs librarymangler.py, which renames every symbol to mmint_* except those matching a glob in protected_syms.txt (mmwlan*, mmhal*, …). So esp_mesh_ccm_selftest shipped as mmint_esp_mesh_ccm_selftest and app code could not link it. Fix: add one line to protected_syms.txt — the sanctioned way to widen morselib's public surface, not a local hack. It is a change inside mm-iot-sdk, so it is part of the migration's morselib delta.

Phase 2 — the migration crux: fork mm-iot-sdk, or keep vendoring?

Upstream esp-halow keeps mm-iot-sdk as a separate submodule repo, but our morselib work lives inside it. So: (a) fork MorseMicro/mm-iot-sdk and move the work there (matching upstream), or (b) keep vendoring and deliberately diverge? The evidence decides it.

✓ The graft is clean — and provable

The two subtrees our patches touch are git tree-object identical to upstream's public mm-iot-sdk tag 2.10.4 (095b1290):

framework/morselib     68ddbd8d: adef0b13…  ==  095b1290: adef0b13…
framework/src/hostap   68ddbd8d: 9749d630…  ==  095b1290: 9749d630…

The tree our patches were authored against is the upstream tree object — so a reparent applies with zero conflicts. And upstream's own SDK release history (2.6.4→2.10.4, SHA-identical to GitHub) is already a second orphan lineage in our repo, so the graft onto 095b1290 is possible offline, today.

Recommendation: (a) fork mm-iot-sdk, via a phased path (land the esp-halow reunification first with the SDK still vendored, prove Phase 1 green, then extract the SDK as a second change). The work is essentially all SDK work; vendoring buries it with no shared history and turns every future bump into an un-bisectable file drop. The tradeoff is honest: (a) is two forks + two submodule pointers to keep coherent; (b) is one repo at the cost of permanent divergence. As long as stack bumps are on the roadmap, (a) wins. Note both target forks already exist and are unused — teapotlaboratories/esp-halow and teapotlaboratories/mm-iot-sdk (created 2026-06-25) — so the migration pushes branches to forks it does not have to create.

The port-forward: 2.10.4 → 2.12.3 concentrates in one file

Most of our 8452 lines ride nearly free — umac/mesh/ (3280) and umac/ibss/ (687) do not exist upstream at any tag; ccmp.c is byte-identical across 2.10.4 / 2.11.2 / 2.12.3; umac_twt.c is ours and upstream barely touched it. The risk concentrates where both sides rewrote heavily:

Structural complication: upstream split the datapath (umac_datapath_sta.c) and added a full umac/relay/ — 1370 lines at the 2.12.3 target, not the header-only stub it is at 2.11.2. Our datapath work must be re-sited across the new AP/STA split, not just conflict-resolved. Land 2.11.2 first for a bisectable midpoint.

⚠ A bench hazard the port introduces — pin it first

2.11.2 flips CONFIG_HALOW_PS_MODE's default false → true (gated on MM_WAKE/MM_BUSY ≥ 0, which our board overlays satisfy). Nothing in our tree sets it, so after the port it silently becomes true — the mmhalow.c:199 PS-disable stops compiling and the shim starts driving WAKE / taking a BUSY IRQ. On board0/board1 those pins are physically unwired, so this turns PS on against pins that go nowhere and moves every PS/TWT number on the bench. Mitigation: set CONFIG_HALOW_PS_MODE=n explicitly in both board defaults before porting — a deliberate decision instead of a side effect.

What a SHA rewrite breaks — the archive tag

⚠ archive/fix1-implementation is 8 commits OFF main

The tag (391eb528) branches from dbecb4dd, not from main — its 8 commits are the pre-squash A-MPDU/defrag/D1 history. A subtree-split of main alone will NOT carry it. filter-repo rewrites all refs it is given, so the archive ref must be present in the working repo when the rewrite runs; after it, verify the tag still resolves and still holds all 8 commits. The brief says it must survive — this is a hard checklist item, not a nicety. rimba's components/halow gitlink and the ESP-IDF component paths (files move components/halow/… → components/halow/halow/…) move with the rewrite too; T0 is the acceptance test for the path changes.

What the suite still does not cover

With board2 online (tools/ppk2_hold.py) all 8 T2 feature tests are automated, so the earlier "defined but not automated" gap is closed. What remains genuinely out of scope, and why:

  • Power-save doze depth — needs the PPK2 current ladder, which is a measurement, not a pass/fail gate.
  • AID ≥ 64 needs 64+ associated STAs — structurally unverifiable on a 3-board bench, recorded as such, not skipped silently.
  • HW-crypto mesh forwarding — a firmware limitation (SW-CCMP is the shipping default), not a regression target.
  • On-air byte-diff vs Linux (the gold-standard frame check) is a separate manual capture the suite points at but does not replace.

Self-review — the harness reviewed its own author

A 6-agent adversarial review (3 reviewers × a verify pass) over the harness Python, the firmware, and the plan found one genuine HIGH bug: the default make test-t1 would have false-FAILed on regtest-swccmp — a radio=False app judged against rimba-hello's banner — gating a healthy tree. (Not hit in this session because T1 always ran with an explicit --app.) Fixed: T1 excludes the regtest-* apps (they are T2's), re-verified on board0. Lower-severity fixes: T1 now asserts the country matches the board overlay (not just ≠ "??"), flags an absent MAC line, and T0's XFAIL is signature-checked so a new proto1 break can't hide under it. The review also caught a plan error — teapotlaboratories/mm-iot-sdk already exists as a fork, so the plan's "does not exist yet" was corrected. The plan's load-bearing numbers all survived the independent re-check exactly.

✓ Net

The suite is green — T0 build (21/21, no hardware), T1 bring-up (14/14 radio apps), and all 9 T2 on-air feature tests pass on the 3-board bench; the one pre-existing break (proto1, missing BCF) is quarantined as a documented XFAIL. This is the baseline the migration must not regress. Live results: docs/regression/rimba-regression-results.md.
