Mesh A-MPDU S0 firmware go/no-go spike
the MM6108 does aggregate 4-addr mesh QoS-data on air

Stage 0 = GO, both halves. A throwaway hardware spike to answer the two questions that gate the whole 802.11s mesh A-MPDU feature: (a) does the firmware advertise A-MPDU capability for a mesh vif, and (b) will the chip emit a real on-air A-MPDU for 4-address mesh QoS-data if the host force-marks the frames eligible? Answer to both: yes. The mesh vif reports AMPDU_cap = 1, and the capture shows genuine multi-MPDU PPDUs — 3 subframes with consecutive sequence numbers sharing one PPDU timestamp, up to 7 MPDUs deep in the return direction.

✓ The result in one line

The MM6108 firmware (fw 1.17.8) does assemble on-air A-MPDUs for 4-address mesh QoS-data when the host marks the frames eligible. So A-MPDU is a firmware capability the host only has to enable — not a firmware change. Temp spike code reverted; both boards back on rimba-hello; sniffer wlan1 down; nothing committed.

Why now — the crypto ceiling just moved

Today the mesh datapath sends single MPDUs: one PPDU per frame, each paying full preamble + IFS + backoff + ACK overhead. Until yesterday the other ceiling was host-side SW-CCMP crypto cost; the just-merged bulk-DMA AES-CCM work (esp_mesh_ccm_ae/ad, ~14–28× faster crypto under load) removed it. With crypto no longer the bottleneck, per-frame airtime is the throughput limiter, and A-MPDU aggregation is the lever that amortises it across many subframes in one PPDU.

A prior feasibility study rewrote the stale blocker-map. Its two load-bearing claims, which S0 exists to confirm on hardware:

  • The CONNECTED gate is NOT the blocker. The mesh datapath's get_sta_state returns CONNECTED unconditionally, so the aggregation eligibility check's first gate is already satisfied for mesh frames. (The old blocker-map claimed a MMWLAN_STA_CONNECTED gate blocked it — stale.)
  • A-MPDU is assembled by the MM6108 FIRMWARE, not the host. The host only (1) sets a per-frame "A-MPDU eligible" flag, (2) runs the Block-Ack session handshake, and (3) reorders on RX. The chip does the subframe packing.

If both hold, the feature is a bounded host-side project, not a firmware change — and S0 is the cheap experiment that proves the firmware side before committing to it.

S0a — capability probe: AMPDU_cap = 1 on the mesh vif

Method. A temporary printf in morselib, immediately after the per-vif capability fetch — umac_interface.c:321, mmdrv_get_capabilities(data->vif_id, &data->capabilities) — printing the two values the aggregation eligibility check reads, plus the vif type. Built rimba-halow-mesh-perf, flashed board1 (efuse MAC e0:72:a1:f8:f9:40, /dev/ttyACM1), read the boot log over serial.

S0PROBE vif=0 type=?? AMPDU_cap=1 ampdu_enabled=1

The type=?? is the tell, not a bug

umac_interface_type_to_str has no case for UMAC_INTERFACE_MESH (= 32), so it falls to default: "??". NONE / STA / AP / IBSS all have named cases — so the only vif that prints ?? is the mesh one. This line is unambiguously the mesh vif, and it reports AMPDU_cap = 1 with global ampdu_enabled = 1.

Consequence for the datapath: the eligibility check's second gate, MORSE_CAP_SUPPORTED(..., AMPDU) at umac_datapath.c:2044, passes for a mesh vif. Combined with the CONNECTED gate already passing, both gates a mesh frame must clear are open in firmware/config today. S0a verdict: GO.

S0b — on-air emission: the chip aggregates mesh QoS-data

The force (temporary, reverted)

To isolate the single question "will the chip aggregate mesh data if told to" from the missing BA handshake, a temporary force in umac_datapath.c (~:2272, inside umac_datapath_process_tx_frame, gated on tx_is_mesh_frame):

  • unconditionally set MMDRV_TX_FLAG_AMPDU_ENABLED on mesh unicast data, and
  • force tid_max_reorder_buf_size = 8

i.e. make mesh frames A-MPDU-eligible without the BA session that would normally precede aggregation. Not a shippable path — a probe of the silicon's willingness to pack subframes.

Rig — secured mesh (SAE + PMF + host SW-CCMP), two direct peers + a dedicated sniffer

rimba-halow-mesh-perf runs args = {0} → args.security_type = MMWLAN_OPEN, but the mesh ignores that: morselib gates mesh security on the compile-time MMWLAN_MESH_SEC_PHASE1, which defaults to 1 (umac_mesh.c:439-440) and sets every mesh peer stad to MMWLAN_SAE / MMWLAN_PMF_REQUIRED (umac_mesh.c:607) with the SAE password hardcoded "rimbamesh2026" (umac_mesh.c:81). So peering did a full SAE+AMPE handshake and the data frames are CCMP-encrypted — on-air, board1's QoS-Data is 100% wlan.fc.protected = 1.

Traffic — saturate the TX queue so frames can back up

board1: iperf -c 10.9.9.136 -u -b 2 -t 14 — UDP, 2 Mbit/s offered (~10× the real link) for 14 s. Over-offering keeps the TX queue non-empty: A-MPDU can only aggregate frames already queued when a PPDU is built, so the queue has to back up for any aggregation to happen.

Detecting aggregation without a radiotap A-MPDU field

The morse0 radiotap header does not expose an A-MPDU status field to tshark, so aggregation can't be read off a flag. It was detected instead by TSFT clustering + consecutive sequence numbers:

  • MPDUs transmitted inside one A-MPDU ride in one PPDU, so they share the exact same radiotap.mactime (the µs TSFT of the PPDU).
  • A genuine aggregate additionally carries consecutive wlan.seq — the MAC hands the queued frames to the aggregator in sequence order.

So: group QoS-Data by identical TSFT; any group of size ≥ 2 is one multi-MPDU PPDU; consecutive wlan.seq inside the group is the airtight signature. Reusable method for this sniffer.

Conclusion — feature is a GO; Stage 1 is the host BA handshake

  • S0a: the firmware advertises A-MPDU for a mesh vif (AMPDU_cap = 1), so the datapath's eligibility gates are open.
  • S0b: given eligible frames, the chip really packs mesh QoS-data into multi-MPDU PPDUs on air (3 MPDUs board1→board0, up to 7 board0→board1).

This confirms the feasibility study's core claim: A-MPDU assembly is a firmware capability the host only has to enable. The only missing piece is the host-side Block-Ack handshake — the S1 work: routing inbound BLOCK_ACK action frames to the BA state machine, and (for a secured mesh) a PMF exemption so the BA action frames are accepted. Nothing in S0 required a firmware change — which is exactly why the spike ran first.

Caveats — stated honestly

⚠ Caveats + residuals — stated honestly

  • SW-CCMP + A-MPDU composition is demonstrated on-air (positive result). The mesh was secured (see the rig), so the forced A-MPDUs were themselves CCMP-encrypted: the firmware aggregated CCMP-protected 4-address mesh QoS-Data (board1's QoS-Data was 100% wlan.fc.protected = 1). So the composition a real secured mesh needs — SW-CCMP encrypt each MPDU, then aggregate the ciphertext MPDUs into one PPDU — is already shown here, not deferred. Honest residual: aggregation depth was still modest (3–7 MPDUs) and this was an all-ESP mesh.
  • Modest depth (3–7 MPDUs). Two likely limiters, neither a blocker: the 1 MHz S1G A-MPDU duration cap and a shallow TX queue at these rates. Deeper aggregation needs higher offered load and/or wider bandwidth — a Stage-2/3 tuning question, not a go/no-go one.
  • Force ≠ handshake. S0b bypassed the BA session, so the throughput is an optimistic proxy (a real deployment pays BA setup/teardown and receiver reorder). It shows the ceiling exists and is reachable, not the steady-state number.

Bench method gotchas (reusable)

  • iperf -b wants a bare integer in Mbit/s. -b 2 is accepted; -b 2M is rejected by this iperf build.
  • iw is not on chronium's non-login-shell PATH. Invoke it as sudo iw … from a non-interactive SSH command, or it is "command not found".
  • pyserial reset-on-open is flaky on these XIAO ESP32-S3 boards. Drive the REPL with dtr=True, rts=False plus a long pre-send read delay to absorb any boot glitch.
  • No A-MPDU radiotap field on morse0. Detect aggregation by TSFT clustering + consecutive wlan.seq — the standing method for this sniffer.
