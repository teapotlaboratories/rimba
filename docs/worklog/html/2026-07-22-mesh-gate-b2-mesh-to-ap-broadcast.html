Mesh-gate B2: mesh → AP broadcast bridging
the last L2-bridge direction · gate app only · no morselib change

The gate now forwards a mesh node's group-addressed frames down to its AP clients — the symmetric counterpart to B1 (the AP→mesh broadcast branch). This closes the final datapath direction of the 802.11s mesh-gate L2 bridge, so a mesh node can initiate to an AP client, not just reply to one. End-to-end verified on-air (continuous ICMP replies). One honest caveat surfaced — documented, not papered over: a mesh node resolves an AP client from the client's gratuitous ARP, not from its own resolution ARP.

The mesh-gate bridges an 802.11s mesh and a SoftAP on one MM6108. Four datapath directions cover the full L2 bridge; B2 is the one that was still missing:

✓ Scope of the win

B2 is proven for the mesh→AP direction it owns (D's real DHCP broadcast is bridged out to the AP client) and D reaches the STA end-to-end. The only caveat is how D learns the STA's MAC in the first place — and that turns out to be a mesh-node lwIP / lossy-broadcast matter, entirely distinct from B2. Details in §3.

1 · What B2 does — gw_mesh_rx_cb

The gate's mesh-RX ext-cb (gw_mesh_rx_cb, on VIF_STA) now inspects each locally-delivered mesh frame. If the 802.3 dst is group-addressed (eth[0] & 0x01) and the gate has AP clients, it re-emits the frame verbatim onto the AP vif (mmwlan_tx_pkt, VIF_AP), then still delivers it locally — because a bridge floods to all ports:

The datapath already prepended the 802.3 header for both frame shapes B2 must forward:

  • plain mesh multicast → [dst=group][src=mesh_sa]
  • proxied AE_A4 multicast (the B1 / AE_A4 RX work) → [dst=group][src=eaddr1]

Loop-safety (analysed + holds on-air)

B2 fires only on mesh RX; the gate's own AP-inject and mesh re-broadcast are TX and never return to the RX cbs; the mesh RMC + own-SA drop already suppress the gate's own re-broadcasts; and B1 (AP→mesh) fires only on AP RX. So an AP client's broadcast (AP→mesh→other-nodes) and a mesh node's broadcast (mesh→AP→clients) each traverse the gate exactly once — no bridge loop.

2 · On-air verification — 3-node mesh+AP+STA

Bench roster: board0 = GATE rimba-halow-mesh-ap (+ B2) · board1 = D rimba-halow-mesh PING_IP=10.9.9.50 (the mesh node that originates a ping to the AP client) · board2 = STA rimba-halow-sta …NO_PING (a responder-only AP client at 10.9.9.50, so D is the sole initiator).

B2 fires — a mesh broadcast is bridged to the AP

ARP-detail logging was added to the gate bridge path:

// the gate receives D's group frame …
mesh group RX: 350 B from e2:72:a1:f8:f9:40 et=0x0800 clients=1
// … and injects it onto the AP vif — TX ok
B2 mesh->AP bcast: 350 B from e2:72:a1:f8:f9:40 ... et=0x0800

The bridged frame is D's DHCP DISCOVER (IPv4 broadcast, 342 B + 8 SNAP = 350 B; a boot artifact emitted before dhcpc_stop). So the gate correctly takes a real mesh group broadcast and delivers it to the AP side.

End-to-end — a mesh node reaches an AP client

D's serial shows reply from 10.9.9.50 seq=4..N — 12–19 continuous replies, 24–108 ms, after the initial ARP-resolution timeouts. The gate shows the matching unicast flow: 14× S5b mesh->AP: 106 B + 14× S5c AP->mesh: 100 B = the D↔STA ICMP round-trips via TX-auto-proxy / S5b / S5c.

3 · The honest caveat — how D actually resolves the STA

The instrumented logs settle exactly which frame drives the ARP resolution — and it is not B2. In the working case, resolution rides in on the client's own gratuitous ARP (bridged the other way by B1):

⚠ B2 is not the resolver — the client's gratuitous ARP is

A diagnostic that logged every mesh group frame the gate receives showed it gets only D's IPv4 (et=0x0800) group frames — D's own ARP request (et=0x0806) for the STA never reaches the gate. In the one run where D was reset after the STA's gratuitous ARP (so D never cached it), D pinged for 30 s with all timeouts and never emitted a bridged ARP → 0 replies.

Root cause — isolated 2026-07-22

The mesh node's lwIP does not emit a resolution ARP for the AP client on demand. With the STA's periodic gratuitous ARP disabled (CONFIG_LWIP_ESP_GRATUITOUS_ARP=n) so D could only resolve on its own, over 60 s / 27 ping timeouts D transmitted:

  • 2× DHCP DISCOVER (et=0x0800)
  • 1× a gratuitous ARP for its own IP (op=1 spa=10.9.9.100 tpa=10.9.9.100)
  • never a resolution ARP for the STA (op=1 tpa=10.9.9.50) — and 0 replies

So the ICMP to 10.9.9.50 is queued/dropped without lwIP's etharp_query ever firing a request. It is not a datapath-TX problem (D's own gratuitous ARP does go out over the mesh, proving the datapath transmits ARP frames fine) and not a B2 problem (B2 bridges every mesh group frame it receives — there is simply no resolution ARP to bridge).

Deepened — and then revised to flaky, not never

  • D's netif is correct: readback ip=10.9.9.100 mask=255.255.255.0 gw=0.0.0.0 → 10.9.9.50 is on-link (same /24), so this is NOT the ERR_RTE off-link-with-no-gateway drop (etharp.c:826-852). D's own code comment already notes off-subnet replies "die in lwIP ARP" without a gateway — but the STA is on-subnet here.
  • When D is peered (netif up=1), esp_ping's sendto(10.9.9.50) succeeds — 0 send errors, 27 timeouts — so etharp_output→etharp_query IS reached and returns ERR_OK (packet queued pending ARP). Per lwIP (etharp.c:966-981) a fresh EMPTY→PENDING entry means is_new_entry and etharp_request() IS called. Yet the resolution ARP (op=1 tpa=10.9.9.50) never appears on-air, while D's gratuitous ARP (op=1 tpa=10.9.9.100) — a byte-identical broadcast ARP frame differing only in the target IP — does. Both go etharp_raw→netif linkoutput→mesh TX, which is address-agnostic.
  • When D is alone (no peer, netif up=0), sendto fails outright (send error=0) — expected (link down).

🐛 Real nature — an intermittent mesh-broadcast delivery failure

With CONFIG_LWIP_ETHARP_DEBUG on D and the STA's gratuitous ARP OFF: in one run D reached the STA fine (reply from 10.9.9.50 seq=37..50 continuously) — so its OWN resolution ARP DID work that time. In an identical-config re-flash run D got 0 replies / 18 timeouts and the gate saw NO resolution ARP (only D's gratuitous ARP tpa=100). D's ETHARP trace shows ethernet_output: sending packet (frames ARE handed to the link) but the resolution ARPs intermittently don't reach the gate, while the gratuitous ARP + DHCP (fewer, periodic) do.

So mesh broadcast frames are sent No-Ack (unacked multicast, QoS ack policy 0x0020 — umac_datapath.c:2506) and are lossy on-air, so lwIP's few ARP retries hit-or-miss. Passive learning via the client's periodic gratuitous ARP (+ TRUST_IP_MAC) is what makes it usually work. NOT a clean deterministic "no ARP" bug, and NOT a B2 defect. A robust fix = gate proxy-ARP for the mesh subnet, or more ARP retries / reliable-multicast — not more B2.

Why it "works" in practice (and the flakiness)

ESP-IDF's TRUST_IP_MAC lets D create an ARP entry directly from the STA's gratuitous ARP (bridged AP→mesh by B1), which the STA emits every CONFIG_LWIP_GARP_TMR_INTERVAL=60 s. So D reaches the STA once it catches one — up to a 60 s wait, and never if the client is silent (gratuitous ARP off → the 0-reply case above). B2 delivers mesh broadcasts to AP clients (its job); the gap is entirely that the mesh node never issues the on-demand resolution ARP.

4 · Build detour (self-inflicted, resolved, net-zero)

⚠ A tracked sdkconfig.defaults clobbered by a diagnostic — no code kept

Getting the ETHARP trace needed a clean sdkconfig regen, and the link failed with undefined reference to umac_ap_get_beacon. I misdiagnosed this as a latent morselib bug and guarded the AP calls in umac_mmdrv_shim.c — but the real cause was that my diagnostic cat > firmware/rimba-halow-mesh/sdkconfig.defaults had overwritten a tracked file that intentionally sets CONFIG_HALOW_AP_MODE=y ("Mesh reuses the AP-type interface + beacon code path, so the AP-mode morselib sources must be compiled in"), and my rm deleted it. So the mesh app is supposed to build AP-on. Restored the sdkconfig.defaults and reverted the shim guard — net-zero morselib change from this detour.
