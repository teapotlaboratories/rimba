Mesh-gate S2: RANN RX + gate list + Connected-to-Gate beacon
the receive side, on every node · byte-exact port of Linux hwmp_rann_frame_process

Second stage of the approved 802.11s Mesh-gate port. Where S1 made an ESP emit a Root Announcement, S2 is the receive side that runs on ALL nodes: a node that hears a RANN learns a path back to the root, records it as a gate when RANN_FLAG_IS_GATE is set, kicks a root-confirmation PREQ to build a real forwarding path, re-floods the RANN (TTL−1, hop+1, accumulated metric), and advertises "Connected to Mesh Gate" in its beacon. Follow-Linux and byte-diffable — a port of net/mac80211/hwmp_rann_frame_process (kernel 6.12.21, the bench interop tree).

✓ All three S2 deliverables PASS on-air

An ESP node (test-mesh-gate-rx) peers with a live Linux gate, learns the gate from its RANN, re-floods that RANN byte-correctly, and advertises the Connected-to-Gate beacon bit — all three captured and machine-checked against the Linux transmission. Bonus: the ESP's root-confirmation PREQ reached the gate and built a bidirectional active path.

Goal

Second stage of the approved rimba-mesh-ap-mesh-gate-discovery-design.md port, following S1 (gate RANN TX, verified 2026-07-21). S2 gives every node the receive behaviour: hear a Root Announcement → learn a path back to the root, record it as a gate on RANN_FLAG_IS_GATE, kick a root-confirmation PREQ to turn that into a real forwarding path, re-flood the RANN with the standard transform, and set the "Connected to Mesh Gate" formation-info bit. Everything derives line-for-line from net/mac80211/hwmp_rann_frame_process so each on-air frame is byte-diffable against a live Linux transmission.

What was implemented

components/halow, uncommitted (git -C components/halow diff). umac_mesh.c gained ~187 lines for S2 (combined S1+S2 = 264 insertions); umac_mesh.h +6 for S2.

mesh_path_entry — gate/root fields

is_root, is_gate, rann_snd_addr[6], rann_metric, last_preq_to_root_ms — the same fields Linux carries on struct mesh_path. The memset-zero init leaves every path non-gate / non-root by default.

RANN RX handler — the hwmp_rann_frame_process port

  • Parse flags / hop / ttl / rann_addr / seq / interval / metric; hopcount++.
  • Ignore our own RANN.
  • Require the sender be a known peer (== Linux sta_info_get).
  • new_metric = orig_metric + last_hop (overflow-clamped).
  • mesh_path_get_or_add.
  • Freshness gate: !MESH_SN_GT(orig_sn, sn) && !(sn==orig_sn && new_metric<rann_metric) — the exact Linux !SN_LT(mpath->sn, orig_sn) && !(…).
  • On accept: update sn / rann_metric / is_root / rann_snd_addr / expiry_ms.
  • Root-confirmation PREQ via umac_mesh_start_discovery, gated to MESH_ROOT_CONFIRM_INTERVAL_MS = 2000.
  • mesh_path_add_gate when the frame is a gate.
  • Re-flood: TTL≤1 → stop; else TTL−1, hop+1, new_metric, original addr/sn/interval/flags kept, via umac_mesh_tx_hwmp.

Gate list

  • mesh_path_add_gate — idempotent, g_mesh_num_gates++ (== Linux -EEXIST / num_gates++).
  • mmwlan_mesh_gate_count() — public accessor (== mesh_gate_num).
  • Eviction decrement in mesh_path_get_or_add; reset in mesh_path_tbl_reset.

Beacon Mesh Config IE — now dynamic

mesh_formation_info_byte + mesh_capability_byte (forward-declared before mesh_build_config_ie):

  • Formation Info = (neighbors<<1) | connected_to_gate(bit0) (== Linux mesh_add_meshconf_ie).
  • Capability = ACCEPT_PLINKS(0x01) | FORWARDING(0x08 if multihop).

3 · Beacon Connected-to-Gate bit

board0's S1G beacon Mesh Config IE: Formation Info = 0x03 → bit0 Connected-to-Gate = 1, peer_count (bits 1–6) = 1 — byte-identical to the Linux gate's 0x03. Capability = 0x09 → ACCEPT_PLINKS(0x01) + FORWARDING(0x08).

The Linux gate additionally sets bit4 0x10 → cap 0x19; that bit is out of S2 scope and is not compared in peer matching, so the small difference is expected and harmless.

✓ Bonus — bidirectional path confirmed

chronite's iw dev wlan1 mpath dump shows an ACTIVE path to e2:72:a1:f8:ef:a4 (metric 2410, hop 1) and station dump shows it ESTAB — the ESP's root-confirmation PREQ reached chronite and built the return path.

Static review + two fixes

✓ Independent static verification — MATCH on every lens, zero blockers

A 4-lens adversarial workflow (rx-parse-reflood · freshness-sn · gate-list-beacon · regression-loops) plus synthesis confirmed: the RX offsets vs struct ieee80211_rann_ie, the freshness SN math (proven identical including wraparound), the re-flood byte string, the gate-counter integrity, the beacon bits (no peering regression — mesh_matches_local ignores those bytes), and no re-flood ping-pong loop (freshness + own-RANN ignore + TTL terminate it).

Two workflow-surfaced risks were fixed, then re-verified on-air — both behaviour-neutral for the ESTAB scenario, no regression:

  • (a) Sender check relaxed from != ESTAB → drop to not-a-known-peer → drop (== Linux sta_info_get: a non-ESTAB peer is processed at MAX metric, not dropped).
  • (b) Expiry set on accept — expiry_ms = now + MESH_PATH_LIFETIME_MS on an accepted RANN, so a gate path learned purely from RANN isn't left at expiry_ms=0 (the first eviction victim).
