Stress-testing the harness + a serial-capture retry hardening
One T1 run had failed on a lone serial-capture glitch that vanished on retry (serial capture failed: device reports readiness to read but returned no data … multiple access on port?). Instead of waving it away, stress-tested the harness itself — looped the hardware-touching tiers with NO retries to measure the real flake rate. Clean-rail: 0 flakes across the first 8 T1 runs (~96 captures). Then flakes appeared after a dscycle iteration disturbed the PPK2 rail, on DIFFERENT random apps each time (rimba-halow-sta, rimba-halow-mesh-ap, even the radio-free rimba-hello) — ~4 flakes across ~168 board2 captures = ~2–3% per capture, ~1 in 4 runs, and NOT app-specific. Root cause: board2 is PPK2-powered, and a brief rail wobble resets the ESP → its USB-serial-JTAG drops and RE-ENUMERATES mid-capture (often renumbered), which pyserial surfaces as readable-but-0-bytes; bus-powered board0/board1 showed 0 flakes. The harness treated this transient as a hard gating FAIL. Fix (tools/regtest/common.py): a shared retry-aware _capture() + _await_port() — on a mid-capture SerialException re-resolve the port by efuse MAC and recapture a fresh window; exhausting the retry re-raises so a genuinely dead board still FAILs; the retry is LOGGED not silent. efuse_mac threaded through t1_smoke.py:159 + t2_onair.py:152/236/345. Verified: unit test covers both the recovery and persistent-fail branches; 10 post-fix runs = 120 captures, 0 fail (no regression). HONEST CAVEAT: a live transient never fired in the post-fix window (even on a dscycle-disturbed rail), so the fix catching a REAL event on-bench is unobserved — only the mock holds. The first buggy edit (a 'bench' name out of scope → 10/12 fail) was caught by the bench, prompting pyflakes in the loop. Incidental: dscycle's pgrep -f ppk2_hold.py | kill -9 is broad enough to kill any process mentioning the holder. Bench left radio-silent; changes uncommitted (weekday window).

HWMP path-flapping — a forced-topology TEST ARTIFACT, not a bug
A 3-lens adversarial comparison against the pinned Linux net/mac80211/mesh_hwmp.c settled a long-standing question: there is NO production HWMP flapping bug. The anti-flap logic is a faithful Linux port (mesh_path_update at umac_mesh.c:1984 == hwmp_route_info_get: stale-SN reject + adopt only via same-next-hop OR a 10/9 ~11% metric-hysteresis win, applied independently of the SN test), so two genuine similar-metric paths cannot flap. The ESTAB metric gate already exists too — mesh_last_hop_metric (:2339) returns MESH_METRIC_MAX for a non-peer transmitter, exactly like Linux airtime_link_metric_get, so a route learned via a non-peer loses the compare. The 07-11 oscillation (~319 RA=board0 correct-relay vs ~221 RA=board2) was manufactured by the bench-only peer allowlist (mesh_peer_allowed:517, allows-all when count==0, i.e. off in production), which creates an "in-RF-range + HWMP-heard + not a data peer" state that cannot occur in an auto-peering mesh. The proximate RA-flip is the origin RA=dest fallback (umac_datapath.c:3761), not route-learning. Two real, separate residuals recorded: D1 (small — implemented + bench-verified the same day: drop + discover instead of blasting an unreachable RA=dest; single-hop 0.46 Mbit/s regression-clean, multi-hop ramps to 1.66) and D4 (bigger — no tx-status→fail_avg→LINK_FAIL_THRESH(95)→mesh_plink_broken self-heal, so a silent black-hole never heals; would NOT have cured the 07-11 flapping since allowlist-dropped frames hardware-ACK above the drop).

#20 resolved — the RELAY's crypto mode is the discriminator
Settled #20 with a clean single-variable A/B and then proved the mechanism offline. The discriminator for whether a forwarded multi-hop mesh frame is delivered is the FORWARDING RELAY's crypto mode — NOT the receiver's host stack. With a native HW-crypto Linux receiver (chronite, not recompiled) and only the ESP relay's g_mesh_sw_crypto flipped: a SW-CCMP relay's forwards ARE delivered by the Linux kernel (33 frames = 31 ICMP + 2 ARP, 31/31 replies), while a HW-crypto relay yields 0 delivered despite ~30 forwards on air — with a 28/28 positive control in both rows. That refutes BOTH earlier verdicts, which were relay-crypto-confounded: the 07-11 "universal MM6108 FW limitation (Linux also withholds)" AND the 07-14 "ESP-vs-Linux receiver difference" reversal. MECHANISM RESOLVED by an offline CCMP-MIC verify: board2's own MGTK was printed at AMPE install, its forwarded group frames captured byte-exact on chronium's morse0, and MIC-verified offline via a Python port of morselib ccmp.c mesh_ccmp_aad_nonce — the SW-relayed forward MIC-VERIFIES (plaintext = valid ARP, 2/2) while the HW-relayed forward MIC-FAILS (3/3). So the MM6108 HW-crypto path emits a MALFORMED forwarded frame that any canonical receiver rejects; morselib/morse_driver RX are exonerated and the fix is FIRMWARE. Verified for GROUP (MGTK); unicast (MTK) presumed the same. Practical answer unchanged: host SW-CCMP forwarding works end-to-end and stays the shipping default.
