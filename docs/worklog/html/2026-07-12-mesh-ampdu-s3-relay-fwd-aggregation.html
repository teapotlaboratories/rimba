Mesh A-MPDU S3: relay forward-leg aggregation + FIX-2
route the forwarded unicast onto the aggregating data path — and stop it blocking the evtloop

Code done, builds green, adversarially reviewed (3 findings found + fixed), not yet bench-verified. The relay's forwarded unicast now rides the per-TID data path (DATA-class alloc + next-hop BA session + AMPDU_ENABLED) so the firmware A-MPDUs it — the dominant multi-hop-goodput lever (B6). At the same seam, FIX-2 makes the forward non-blocking, removing the 1000 ms evtloop stall that armed the relay interrupt-WDT crash. Mirrors net/mac80211 ieee80211_rx_h_mesh_forward (re-inject into the data TX path, TID preserved). Uncommitted, stacked on S1/S2.

✓ Why the data-class alloc is the decisive move

MMDRV_PKT_CLASS_MGMT routes to the mgmt_q (pageset.c:202, tx_mgmt_handler) which never A-MPDUs; MMDRV_PKT_CLASS_DATA_TIDx routes to the aggregating tx_data_handler. Today's forward is built mgmt-class, so it can't aggregate no matter what flags it carries. The new build_mesh_data_frame allocs it DATA-class; the rest (seqno, BA, AMPDU flag, channel) all line up at TID 0.

1 · What shipped

FIX-2 and S3 land together at the forward seam. A compile-time toggle MESH_FWD_DATA_AGGREGATE (default 1) gates the S3 aggregation for bench A/B; FIX-2's non-blocking behaviour is unconditional (you never want to toggle back into the crash).

FileChange
frame_constructor.c + frames_common.hNew build_mesh_data_frame() — allocs on MMDRV_PKT_CLASS_DATA_TID0+tid instead of MGMT.
umac_mesh.hToggle MESH_FWD_DATA_AGGREGATE (=1) + MESH_FWD_TID (=0).
umac_mesh.cumac_mesh_forward_data allocs DATA-class when the toggle is on.
umac_datapath.cumac_datapath_tx_mesh_keyed_frame: FIX-2 non-blocking gate at the top; S3 per-TID seqno + aggr_check + tid 0 + AMPDU-eligibility for the unicast forward; channel routing.

The unicast aggregation retag (mirrors the local-origin path)

// for a forwarded UNICAST (key_type == PAIRWISE; stad = next-hop peer):
seqno  = sta_data->tx_seq_num_spaces[0]        // per-TID, on the next hop (not baseline)
umac_datapath_aggr_check(umacd, stad, 0, ssc)  // open BA session on the next hop
tx_metadata->tid = 0
tx_metadata->flags |= AMPDU_ENABLED  if umac_ba_is_ampdu_permitted(stad, 0)
mmdrv_tx_frame(txbuf, !fwd_aggregate)          // is_mgmt=false → DATA channel (aggregates)

Group re-broadcast (key_type == GROUP) is untouched: baseline seqno, no aggregation, MGMT channel. SW-CCMP-per-MPDU is preserved and composes with A-MPDU (validated in S0/S1). The next-hop-keyed seqno is not just correct but required — TA is always our mesh MAC, so the next hop keeps one (TA=us, TID 0) reorder window that local-origin and forwarded traffic must share.

◑ Discovered during this pass — the forward runs in the evtloop, not bus_lock

Verifying the review corrected the interrupt-WDT root-cause worklog: umac_datapath_rx_frame queues the RX frame + wakes the umac-core evtloop task (:1945-1947) — the forward runs there, holding no bus_lock. The blocking forward stalls the evtloop, not the SPI worker. Verdict + fixes unchanged; the root-cause worklog's Stage 1 was corrected.
