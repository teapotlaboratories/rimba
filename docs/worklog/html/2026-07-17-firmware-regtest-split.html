2 · The key finding — rimba-halow-sta was not a STA example

🐛 A user-facing "STA example" that was actually a PPK2 measurement rig

rimba-halow-sta's header read "TRIGGERED power-save ladder (C6-harness-ready, netif-free, zero traffic)" — it associated then ran a fixed 4-tier PS ladder, marking each phase as D5 GPIO pulses for the PPK2/C6, behind a flash-hold guard and a close-bench TX cap (line 153: "Remove for a real deployment"). That ladder was already cloned verbatim into test-power (the tp DUT). Yet the radio-silent runbook + rimba-linux-node-setup.md still told users to flash it as "station, static 192.168.12.2" — a stale mismatch.

So it was removed-and-recreated: a clean join-AP-over-SAE + static-IP + continuous-ping STA (derived from test-apsta-sta minus the TEST| hooks). This filled the missing "clean STA example" and fixed those stale docs at the same time.

3 · Method — two workflows

The decision surface was 17 non-regtest apps × (is-it-a-fixture? · example-worthy? · superseded by a regtest test? · what edits keep the suite green?), so it was fanned out:

  • 19-agent analysis workflow — one agent per app reported test-scaffolding with file:line, example-worthiness, the superseding test-* test, and the exact manifest.py/Makefile/docs edits each choice needs; plus a lineage agent (mapped every test-* app to the rimba-* it was cloned from, confirming each carries disqualifying harness hooks) and a doc-reference agent (split every doc mention into living = edit vs historical worklog = leave alone).
  • Owner picked all four dispositions (each = the recommended option): git rm the removals · strip scaffolding + a README per survivor (no renames this pass) · keep a cleaned ap-perf + sta-perf iperf pair and git rm mesh-perf · clone hello → test-idle.
  • 11-agent cleanup workflow — one agent per surviving example stripped the cited scaffolding (removing dead callers/decls/includes together), grep-verified no removed symbol remained, and wrote the README.
