Mesh host-fragmentation: on-air bench verification
the committed fix is broken — the MM6108 firmware re-processes host-built fragments

Bench-verified the committed but untested host 802.11 fragmentation fix for mesh SW-CCMP (fragment→encrypt, halow 9c7daabd). Verdict: NOT verified — it is broken on this firmware. The first fragment goes out perfectly; every fragment past it is corrupted by the FW. A control run with the fix disabled (A-MPDU sends full-size frames whole) delivers 100+ pings end-to-end, proving the rig is sound and isolating the failure to the fragment path. A follow-Linux source cross-check (both trees) confirms this is a hard firmware wall in the #20 family, not a fixable morselib flag.

The test

A forced all-ESP line, board2 as the relay (it is the only fully-wired ESP DUT). board0 large-auto-pings board1 (1400 B, 200 ms) through board2; a temp toggle forces mesh unicast non-A-MPDU so every full-size mesh SW-CCMP frame deterministically takes the host fragmentation path. RX/TX counters (g_frag_dbg[]) dump over the console; chronium's morse0 monitor captures the frames off-air.

Test A — force fragmentation: FAIL

✗ The relay MIC-fails ~2 of every 3 fragments and never reassembles

board0 fragments correctly (TXorigin=84, TXsent=252 = exactly 3 fragments/ping). But the relay board2 reads RXmesh=248 RXfrag=238 mic_fail=158 decrypt_ok=89 reasm=0 fwd_reached=0 — it decrypts only ~36% of fragments, reassembles none, forwards nothing. board1 gets no traffic; pings all time out.

The ~2 : 1 fail : ok ratio is the tell: a 3-fragment frame = 2 full fragments + 1 small remainder; the 2 full ones fail, the small one decrypts. The on-air capture shows exactly why.

On-air root cause — fragments i > 0 are double-headered

Decoding the 4-address QoS-Data mesh frames off morse0 (raw hex, one ping = seq 421):

✓ correct frag 0  ·  len 731  ·  MoreFrags=1
MAC hdr32 B · 4-addr QoS
CCMP8 B · PN@ off 32
ciphertext + MIC687 B
✗ broken frag 1  ·  len 763  ·  MoreFrags=0 · Retry=1
MAC hdr32 B
DUP MAC hdr+32 B · byte-identical
CCMP8 B · PN@ off 64
ciphertext + MIC687 B
✗ broken frag 2  ·  len 168  ·  MoreFrags=0
MAC hdr32 B
DUP MAC hdr+32 B
CCMP8 B
cipher + MIC92 B

Every fragment with fragment_number > 0 carries a byte-identical duplicate of its own 32-byte MAC header, pushing the CCMP header to offset 64. The host encrypt (umac_datapath_sw_ccmp_encrypt) emits a correct single-headered frame, and the builder code is identical for every fragment index — so the extra header is added after host encryption, on the FW/TX path, and only for frag# > 0 (frag 0 is untouched; the FW also sets Retry and clears MoreFrags). The prepended header corrupts the CCMP AAD/length the receiver reconstructs → MIC fails → the fragment set never completes → no reassembly → no forwarding.

Test B — toggle off: A-MPDU sends full-size frames whole: PASS

Same rig, fragmentation forced off. This is the control that validates the rig and shows the practical path works.

✓ Recovery is unambiguous

At cold start the fix fires naturally (the BA session isn't up, so ampdu_eligible=false) — board0 fragments 2 frames and those fail exactly as in Test A. Once the BA session establishes, TXorigin freezes at 2: fragmentation stops, frames go whole/aggregated, board2 forwards them (fwd_reached 0→223), and board0 gets 100+ replies. reasm stays 0 throughout — whole frames need no reassembly.

This also exposes the fix's real problem: the residual case it exists for — cold-start / pre-BA full-size frames — is exactly where its fragments get mangled and lost. It neither helps nor hurts versus letting the FW fragment them; it just adds a broken path.

Follow-Linux cross-check — it's a firmware wall

The morse Linux driver runs the same MM6108 + 1.17.8 firmware. Does it host-fragment successfully where the ESP can't? No — Linux never host-fragments on this firmware, and neither stack has any per-frame "already-fragmented / leave-it-alone" flag:

FactEvidence (grep, both trees)
mac80211 skips host fragmentation under SUPPORTS_TX_FRAG; defaults to DONTFRAG net/mac80211/tx.c:967-968, 964, 1275-1280
SUPPORTS_TX_FRAG set straight from the FW HW_FRAGMENT capability bit morse_driver/mac.c:6835-6836; cap capabilities.h:86
Fragmentation is FW-owned via a global threshold command — host sends whole MPDUs mac.c:5632 → command.c:2387; morselib mirror umac.c:1598
No fragment / raw / seq field in the host→FW TX descriptor — either tree Linux skb_header.h:243-251,57-71 · morselib mmdrv.h:609-619

Crypto mode — verified live + source (2026-07-14). The bench Linux nodes run HW crypto (no_hwcrypt=N on chronite/chronium/chronosalt; set_key HW-offloads CCMP, mac.c:5141-5230). Under HW crypto the FW does fragment→encrypt in the correct order — but #20 blocks the HW-crypto multi-hop forward, so default Linux mesh is single-hop-only for encrypted traffic and there is no working multi-hop-encrypted Linux mesh. So there is no Linux precedent for the ESP's case (SW-CCMP multi-hop). And mac80211 never host-fragments by default (frag_threshold=-1 → DONTFRAG on every unicast). Forcing Linux to SW crypto (no_hwcrypt=1) for multi-hop would hit the same wall. (An earlier "Linux uses HW crypto to sidestep this" phrasing was imprecise and self-contradictory vs #20 — the accurate statement is that the ESP operates where Linux does not, so there is no reference and no host-side fragmentation fix on this firmware.)

✓ On-air confirmed (2026-07-14) — under HW crypto the FW fragments cleanly and delivers

Secured single-hop Linux mesh chronite↔chronogen (HW crypto), forced MCS0 + a 512 B frag threshold → the FW fragmented a 1400 B ping into 4 clean fragments: MoreFrags 1,1,1,0, incrementing frag#, constant seq#, each 528 B with its own CCMP header + consecutive PN (82,83,84,85). Raw hex of frag=1: CCMP header at offset 32 (58 01 00 20 …) = single-header — vs the ESP's frag>0 double-header (CCMP at offset 64, len 763). And it delivered 61/70 (87%) — chronogen reassembled + replied — vs the ESP relay's 0%. Same silicon + FW; the only difference is who encrypts: Linux = the FW (HW crypto owns fragment→encrypt, gets it right); ESP = the host (SW-CCMP → the FW re-headers frag>0). This closes the last inference: the "never fragment" (MTU-cap / min-MCS) path is confirmed as the only option for the ESP's SW-CCMP mesh.

Conclusion & recommendation

Host fragment→encrypt is fundamentally incompatible with this firmware

A hard FW limitation in the #20 family, not a morselib logic bug — the encrypt + builder emit a correct single fragment; the FW mangles the multi-fragment burst. Confidence: VERY HIGH on the descriptor-level facts, HIGH on the conclusion.

Deterministic path: abandon host fragmentation for mesh SW-CCMP and keep every full-size frame under the single-PPDU limit so it is never fragmented at all — a mesh-vif MTU cap (the host never hands down a frame that would need fragmenting) or a min-MCS floor on the mesh RC (a full frame always fits one PPDU at the worst rate). Either keeps frag#=0 always. Meanwhile A-MPDU-whole already carries the steady-state mesh fine. The committed 9c7daabd host-fragmentation fix should be reverted or gated off — left in the tree for the owner's review, not reverted here.

✓ Revert + ESP verification (2026-07-14) — the revert IS the practical fix; MTU-cap unnecessary

9c7daabd reverted (staged, builds green). Then verified what the reverted mesh actually does — and the "MTU-cap is needed" premise turned out much narrower than the forced Test A/B implied. Measured (Linux, same FW): a full-MTU frame at MCS0/1 MHz goes WHOLE (1566 B) — MCS0 is not where a normal frame fragments; the mesh only fragments when it drops below MCS0 (the 1 MHz repetition mode, single-PPDU budget ~728–740 B — so 728 was right for the worst rate, not MCS0). On the reverted ESP mesh (real conditions, full-size multi-hop): fragmentation is rare — 1 frame in 20 s (~0.24%) vs 410 whole; ~86% ping on a marginal MCS0/MCS2 bench link where the ~14% loss is low-rate RF, not fragmentation; a good link (MCS7) = zero fragmentation. So the revert removes the systematic breakage. An MTU cap is overkill (caps every frame for a <0.3% weak-link edge); a min-MCS floor is the right optional weak-link hardening (future item, not a bolt-on).

Method & disposition

  • Adversarial pre-bench review (4-lens workflow) caught one real counter-attribution bug (mic_fail also hit by the protected-PREP decrypt path) — fixed by gating on the DATA counter space before flashing.

  • Left the bench radio-silent: all three ESPs on rimba-hello (banner + zero mmwlan lines confirmed), chronium + chronite wlan1 down. Temp instrumentation stripped; stripped tree rebuilds green.
