S3 relay forward drop root-cause: host SW-CCMP MIC failure
root cause = pairwise MTK key mismatch (board0 encrypts under a different key than board2 decrypts) · AAD + addresses match

The ESP relay (board2) drops ~99% of forward-intended mesh data frames at the host SW-CCMP decrypt before they ever reach the forward call, so the relay forwards ~0 sustained data and S3's forward-leg A-MPDU can't be measured. The read that was blocked for several sessions is now done: the drop is a CCMP MIC failure (not replay), and it is specific to the multi-hop A4≠TA forward — single-hop SW-CCMP + A-MPDU decrypts 99.4% fine. The host TX chain is exonerated, and the revived on-air capture shows board0's forward frames are valid on the wire. A dual-side CCMP dump then pinned the root cause: board0 encrypts the forward frame under key=6f 1d 9d 9c 48 af d6 49 while board2 decrypts under key=91 bd e3 64 a0 a4 40 5a — the AAD (masked-FC + A1/A2/A3/A4 + QoS) is identical on both sides, only the key differs. The board0↔board2 pairwise MTK is not synchronized, so the MIC never verifies. The crypto recipe and the mesh header addresses were first verified to follow mac80211, so the recipe is not at fault; the open item is now the mechanism — an asymmetric AMPE MTK rekey/install under peering churn.

✓ Root cause found — pairwise MTK key mismatch

The dual-side CCMP dump settles it. Capturing board0's encrypt and board2's decrypt of the same unicast multi-hop frame: the AAD matches byte-for-byte (masked-FC + A1=board2 + A2=board0 + A3=chronite + A4=board0 + QoS) but the key differs — board0 key=6f 1d 9d 9c 48 af d6 49 vs board2 key=91 bd e3 64 a0 a4 40 5a. So the board0↔board2 pairwise MTK is not synchronized → the MIC never verifies → the ~99% forward drop. This overturns the earlier "same correct key both sides" assumption (that was inferred from the correct key_stad selection; the actual key bytes differ). The AES-CCM itself is fine (RFC-3610 KAT passes; single-hop = 99.4% where the same peering had synced keys). The crypto recipe + mesh header addresses were verified to follow mac80211, so the recipe is not at fault. Open item = the mechanism: a leading candidate is an asymmetric AMPE MTK rekey/install under peering churn. Nothing committed; bench left radio-silent.

1 · The drop bucket: MIC failure, not replay

The split instrumentation (temp, in-tree) added FDBG_RX_SW_MIC_FAIL / FDBG_RX_SW_REPLAY_FAIL incremented at the two return false points inside umac_datapath_sw_ccmp_decrypt (umac_datapath.c): the mesh_ccmp_decrypt(...) != 0 MIC-failure return, and the !ccmp_is_valid(...) replay return. The app dump_fwd_dbg prints (mic=.. replay=..). RTC-noinit counters survive the read's warm reset. On the 2026-07-12 multi-hop forward run:

MIC failure2052
replay failure0
decrypt OK26

So it is unambiguously a MIC failure — a key / AAD / nonce mismatch — and not a replay-window reject. decrypt_ok=26 (a few pass) first suggested the passing few were non-aggregated singletons and the failing ~2052 were A-MPDU-aggregated multi-hop-origin frames, i.e. that SW-CCMP might not compose with FW A-MPDU on the S2 origin leg. The single-hop A/B (§2) refutes that composition idea and re-points the cause at the multi-hop forward specifically.

✓ What the A/B proves

Single-hop PLINK-DBG estab=1 close=0 (stable peering) ⇒ the earlier peering-churn idea is REFUTED (stable and works single-hop). SW-CCMP + A-MPDU compose fine under load — the ~1.8 Mbps S1 result stands. The ~99% MIC failure is specific to the multi-hop FORWARD frame (A4≠TA, A3=DA≠A1=RA). The only AAD delta from the working single-hop case is A3=DA=chronite ≠ A1 — so board0 must be putting a different address on-air than the AAD it encrypted over, or the mesh layer rewrites an address after encrypt.

3 · The rig + the forced 2-hop

⚠ Bench-harness gotchas (so the next run is faster)

The forced 2-hop is NON-DETERMINISTIC. All nodes are in RF range; board0 sometimes peers chronite directly and HWMP routes direct → board2 sees ~4 frames instead of ~2000. Root: the app sets board0's peer allowlist AFTER mesh peering can already begin (startup race), so an early chronite peer slips through before the allowlist applies. Workaround: re-boot board0 and retry until board2 mesh_seen ≈ 2000; proper fix = set the allowlist before mmwlan_mesh_start. (The peer-creation paths are all gated on the allowlist already — beacon-open umac_mesh.c:1673, inbound-OPEN dropped on secured mesh, inbound-SAE dropped if no peer — so a mesh_peer_alloc allowlist gate is redundant once the allowlist is set; the hole is purely the timing.)

chronite (Pi5) reboots mid-blast, wiping /tmp (tmpfs) including wpa-interop.conf → the mesh won't come back until the config is re-pushed. Re-push it and bring wlan1 up again.

4 · Host TX chain — exonerated

The header build was traced and the whole host TX chain reviewed against the working single-hop path. Every step is symmetric with the case that decrypts 99.4%:

A1 = next-hop
board2 · umac_datapath.c:3701
A2 = us
board0 · :3702
A3 = DA
chronite · :3703
A4 = SA
board0 · :3706
  • Header build. umac_datapath_construct_80211_data_header_mesh (umac_datapath.c:3673) builds the 4-addr mesh header BEFORE process_tx_frame encrypts, with A1=next_hop (board2), A2=us (board0), A3=DA (chronite), A4=SA (board0). No post-encrypt rewrite — the addr1=bssid clobber at :2030 is in the STA builder ..._header_sta, not the mesh path. The CCMP-header offset is identical single vs multi (both are 4-addr QoS, TO_DS+FROM_DS both set, :3691), and the Mesh Control header is inside the encrypted body (not the AAD, and after the CCMP header) so its size doesn't shift the ccmp_hdr offset.
  • AAD + nonce recipe. mesh_ccmp_encrypt and mesh_ccmp_decrypt (ccmp.c:290/:304) both call the SAME mesh_ccmp_aad_nonce(hdr, ccmp_hdr, …) (ccmp.c:212) — a faithful port of hostap wlantest/ccmp.c ccmp_aad_nonce. The AAD masks out every volatile field: FC subtype/retry/pwr-mgmt/more-data bits (ccmp.c:226/:239), the 802.11 sequence number (:248, keeps only Frag#), and QoS ack-policy/EOSP/A-MSDU bits (:257-260, keeps only the TID). So a normal FW header rewrite during A-MPDU framing (seqno, ack policy) does NOT change the AAD. Unmasked/kept: FC-normalized, A1/A2/A3, A4 (4-addr), TID, and the PN (from ccmp_hdr).
  • Crypto flags. A mesh SW-CCMP frame gets MMDRV_TX_FLAG_AMPDU_ENABLED (umac_datapath.c:2306) but NOT MMDRV_TX_FLAG_HW_ENC (only the else/HW branch sets it, :2291) — so the FW aggregates the already-host-encrypted frame without re-crypting it. Correct SW-CCMP + A-MPDU combination.
  • Key. The key board0 uses for a forward (key_stad=board2) is the SAME board0↔board2 pairwise key that decrypts the working single-hop case.

🐛 Conclusion — the differing bytes are on-air, not in host code

Neither the key, the AAD recipe, the crypto flags, nor the header addresses obviously differ from the single-hop path that works — and it is the SAME chain that decrypts 99.4% single-hop. So the MIC failure is NOT a visible host-code asymmetry: board0's ON-AIR multi-hop frame must differ from the bytes it encrypted the AAD over — an FW-level interaction between the multi-hop-origin 4-address forward and A-MPDU aggregation.

✓ Frames are on-air valid — blank-PN / address / format all ruled out

All 920 frames carry A1=board2 (RA), A2=board0 (TA), A3=chronite (DA), A4=board0 (SA) — confirmed multi-hop, no direct. QoS ctrl 00 01 on every frame (TID 0 + Mesh-Control-Present). There ARE retransmit duplicates (e.g. 0x04da ×3), but those would be replay fails and replay=0, so the MIC failures are not the duplicates. ⇒ the on-air header + CCMP look fully decryptable — the mismatch is invisible on the wire (in what board0 encrypted the AAD over, or the key), not a malformed frame.

💡 Hypothesis that motivated the next run — per-link MTK key mis-selection refuted §6

In AMPE mesh each peer LINK has its own MTK: the board0↔board2 key ≠ the board0↔chronite key (see the comment at umac_datapath.c:3606 — "the MTK is per link (next hop), not per final DA"). The TX stad is resolved by the DEST (chronite); S2 overrides the crypto key to the NEXT-HOP via key_stad = umac_mesh_get_peer_stad(ra) (:2182) and encrypts with it (sw_ccmp_encrypt(key_stad, …) :2281, key_id taken from key_stad :2206). If key_stad ever fails to override to the next-hop (board2) and stays = stad (chronite), board0 encrypts with the board0↔chronite MTK while board2 decrypts with the board0↔board2 MTK → MIC fail — 99% multi-hop, 0% single-hop (where dest == next-hop, so the key is right regardless). This exactly matches the single-hop-works / multi-hop-fails result. The static code looks correct (ra=A1=board2 from the header builder, get_peer_stad(board2) should resolve) and the on-air A1=board2, so the discrepancy is hidden — it needs instrumentation to confirm which key was actually used. That instrumentation was run (§6) and it clears the key.

6 · key_stad confirmation — the per-link-MTK idea, refuted

To settle whether board0 ever encrypts a multi-hop forward with the wrong link's MTK, temp RTC counters (FDBG_TX_MULTIHOP / NH_NULL / NH_EQ_STAD / NH_OK) were added to board0's process_tx_frame, recording per mesh multi-hop unicast whether key_stad overrode to a distinct next-hop peer stad. board0 RTC after a 2297-frame multi-hop blast:

✕ Refuted — it is NOT the key selection

nh_ok = 2267 (98.7%) means board0 encrypts ~99% of multi-hop frames under the correct next-hop (board2) link key — the same key that decrypts single-hop at 99.4% — yet board2 still MIC-fails ~99% of them. Only 30 (1.3%) fell back to a null next-hop, and nh_eq_stad=0 means the key was never left equal to the dest (chronite) stad. 30 fallbacks is far too few to explain the ~2052 MIC failures. The per-link-MTK mis-selection hypothesis is refuted: board0 picks the right next-hop stad, the AAD recipe is symmetric, and the on-air header is valid — and it still fails.

Reconciled with §7: selecting the right next-hop stad is not the same as installing the right key bytes. The dual-side dump in §7 shows board0 and board2 hold different MTK bytes for the same link — so this run's "same correct key both sides" was inferred from the correct selection, but the actual key material was desynced. The bug is the MTK sync, not the key_stad selection.

⚠ Instrumentation gotcha

board0's VBUS cold-cycle does not reliably register as ESP_RST_POWERON, so the RTC-noinit counters read stale garbage once FDBG_COUNT had grown. Fixed by bumping MESH_FWD_DBG_MAGIC to force a zero on a magic mismatch. (board2's PPK2 power-off drains RTC cleanly, so its counters were fine throughout.)

→ Decisive next step — dual-side byte dump (executed, §7)

With selection, recipe, and on-air all clear, the one thing left un-instrumented is the actual key bytes. Log the exact AAD + key + nonce + MIC that board0 uses to ENCRYPT a multi-hop frame and that board2 uses to DECRYPT it, and diff them — the first differing byte pins it. This dump was run, and it pins the root cause: the AAD matches but the key differs (§7).

7 · Root cause — pairwise MTK key mismatch

The dual-side dump was run with a temp mmwlan_ccmp_dbg_capture (filter fixed to unicast A1 — a first pass caught group frames, incidentally showing chronite's GROUP frames also MIC-fail at board2). It logs, for the same unicast multi-hop frame, what board0 encrypts and what board2 decrypts. Before trusting it, the crypto + addressing were verified to follow mac80211 (chronite ~/halow/rpi-linux).

✓ First: the recipe + addressing follow mac80211 — so it is NOT the recipe

The AAD recipe mesh_ccmp_aad_nonce matches mac80211 ccmp_gcmp_aad (wpa.c:318), and the mesh header construct_80211_data_header_mesh matches ieee80211_fill_mesh_addresses (mesh.c:851: A1=next-hop, A2=SA, A3=DA, A4=SA). Both sides build the AAD and the addresses identically and correctly — so the MIC failure is not a crypto-recipe or address-layout bug.

The capture then shows exactly where the two sides diverge — the key, and only the key:

So the entire AAD — masked-FC, A1=board2, A2=board0, A3=chronite, A4=board0, QoS — is identical on both sides. The one thing that differs is the key bytes. board0 encrypts under one key and board2 decrypts under another → the MIC computed over the ciphertext can never match → the ~99% forward drop.

✓ Root cause — the board0↔board2 pairwise MTK is not synchronized

board0's encrypt key 6f 1d 9d 9c 48 af d6 49 ≠ board2's decrypt key 91 bd e3 64 a0 a4 40 5a for the same link. The pairwise MTK the two peers hold for the board0↔board2 link is desynchronized, so every forward frame MIC-fails at board2. This overturns the earlier "same correct key both sides" assumption — that was inferred from the correct key_stad selection (§6, nh_ok=98.7%), but the actual installed key bytes are different. The AES-CCM primitive is fine (RFC-3610 KAT passes; single-hop = 99.4% where the same peering had synced keys), and the recipe + addressing follow mac80211. The bug is purely the key synchronization.

⚠ Leading mechanism (to confirm) + next step — asymmetric AMPE rekey under peering churn

The single-hop A/B ran with stable peering (close=0) → keys stayed in sync (99.4%); the multi-hop runs ran with churn (estab climbing / close>0). AMPE re-derives the MTK on each re-peer, so a re-SAE / re-AMPE that installs the new MTK asymmetrically desyncs the pairwise key. This ties the forward drop back to the peering instability that opened this thread — the plink-liveness fix (fewer re-peers) should reduce it, but the real fix is a symmetric AMPE MTK install/rekey (follow hostap mesh_rsn / wpa_auth key-install ordering). Next: confirm which key (6f 1d 9d… vs 91 bd e3…) is the stale one — board0's own board2-MTK vs a stale / other-peer MTK — and whether it changes across a re-key, then fix the rekey path.
