Mesh-gate S5b — gate egress: mesh AE → AP client
the mesh→AP leg of the L2 bridge · app-level · 3-node on-air PASS

S5b closes the mesh→AP direction of the gate's L2 bridge: when the gate receives a proxied AE_A5_A6 frame whose final DA (eaddr1) is one of its associated AP clients, it delivers the payload onto its AP vif addressed to that client — the direct use of the S5a mmwlan_mesh_register_ae_rx_cb hook. Written, compiles, and verified on-air on a 3-node mesh+AP+STA bench (PASS). Purely app-level — no morselib change (the components/halow diff stays at 655 insertions from S5a).

✓ Result — the gate bridged a proxied mesh frame onto its AP side

Each proxied AE frame reached the gate → gate_ae_rx_cb (S5a hook) fired → recognized eaddr1 as an AP client → rebuilt a 34 B Ethernet frame (14 hdr + 20 L3, src = 02:..:aa) → injected it onto the AP vif toward the STA (mmwlan_tx_pkt SUCCESS). Goal

Implement the mesh→AP direction of the gate L2 bridge (S5b): a proxied AE_A5_A6 frame arriving at the gate whose eaddr1 (final DA) matches an associated AP client should be delivered onto the AP vif, addressed to that client. This is the first direct consumer of the S5a mmwlan_mesh_register_ae_rx_cb hook. The AP→mesh ingress plus retiring the L3 ip_forward path are deferred to S5c.

What was implemented — S5b

All in firmware/rimba-halow-mesh-ap (tracked, modified). Two pieces:

  • AP-client set: ap_sta_status_cb now maintains g_ap_clients[] — add on AUTHORIZED, swap-remove on UNKNOWN — plus a gate_is_ap_client(mac) membership test.
  • gate_ae_rx_cb (registered via mmwlan_mesh_register_ae_rx_cb): if eaddr1 is one of our AP clients, recover the ethertype from the AE payload's LLC/SNAP (aa aa 03 00 00 00 <ET>), build an Ethernet II frame [dst=eaddr1][src=eaddr2][ethertype][L3], and inject it onto the AP vif (mmwlan_alloc_mmpkt_for_tx + mmwlan_tx_pkt with md.vif=MMWLAN_VIF_AP). Non-blocking, runs in the mesh RX task. morselib re-adds the LLC/SNAP and CCMP-encrypts to the client on the AP downlink.

The datapath through gate_ae_rx_cb, end to end:

The frame transformation the callback performs (the 34 B = 14-byte Ethernet header + 20-byte L3):

# AE payload in  (as received on the mesh vif)
[ aa aa 03 00 00 00 <ET> ][ L3 (20 B) ]
  └──────── LLC/SNAP ───────┘

# Ethernet II out (injected on the AP vif)  = 34 B
[ dst=eaddr1 ][ src=eaddr2 ][ ET ][ L3 (20 B) ]
  └──────────── 14 B header ────────────┘

Coexists with the L3 gateway

S5b coexists with the existing L3 ip_forward: a proxied frame to an AP client takes the AP-inject path; all other mesh RX still reaches the mesh netif. Retiring the L3 path is S5c's job.

Test fixture

test-mesh-ae was extended with build-time MESH_ID= (default rimba-smesh) and EADDR1= (the proxied final-DA MAC) overrides — Makefile MESH_ID=/EADDR1= → TEST_MESH_ID/TEST_EADDR1 with CMake propagation — so a mesh node can point at the gate app (rimba-mesh) with eaddr1 set to the gate's AP-client MAC.

✓ Feasibility gate cleared — mesh+AP concurrency on 2.12.3

The whole run had the gate doing mesh and AP and a mesh peer and an AP client simultaneously — mesh+AP concurrency works on 2.12.3 with the S1–S5a datapath. Previously this concurrency was proven only on the older feat/mesh-ap-concurrency branch; it is now confirmed on the 2.12.3 line, which is the prerequisite for the entire S5 bridge.

Notes / footguns

⚠ The STA's AP-side MAC is the MM6108 MAC, not the efuse MAC

Read bc:2a:33:96:b2:33 from the gate's AP client joined log, not from read_mac (which returns board2's efuse MAC).

  • gate_ae_rx_cb runs in the mesh RX task; the AP inject is non-blocking (no mmwlan_tx_wait_until_ready) so it never stalls RX.
  • The AE payload must be LLC/SNAP-encapsulated to recover the ethertype (real mesh data is).
  • mesh+AP concurrency works on 2.12.3 — the feasibility gate for S5, now cleared.
