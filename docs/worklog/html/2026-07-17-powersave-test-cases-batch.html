#1 · FW-blob version-pin (T0, no hardware)

A build-time tripwire for the exact regression the whole tp tier exists to catch: a silent bump to fw 1.17.9 roughly doubles STA power-save current. T0 now asserts, before a board is flashed, that the pinned blob is 1.17.8 by size (480664 B) + sha256 (ce2702b7…). Verified PASS on the pinned blob, FAIL when pointed at the thin-LMAC blob (mm6108-tlm.bin, 473812 B). Highest value/effort of the batch — cheapest choke point, enforces the whole-bench-1.17.8 rule.

#2 · Assoc-embedded TWT → INSTALLED (T2 twt-assoc)

regtest-twt-assoc-sta requests TWT via mmwlan_twt_add_configuration() before connect (the TWT IE in the assoc request). This is the universal path honoured by both the ESP SoftAP and a Linux hostapd_s1g AP — the mid-session action (the twt test) is ignored by a Linux AP, so this is genuinely new coverage.

🐛 an audit caveat, refuted on-bench

umac_twt_add_configuration only stores the config, and umac_twt_process_ie needs a pre-existing agreement — so an earlier audit pass claimed the assoc-embedded path has no INSTALLED accessor. It DOES: mmwlan_twt_agreement_installed(0)==1 after an assoc-embedded setup, against both the ESP AP and chronite's hostapd (flow 0, associated in 3.5 s). The reporter scans flows 0..3 to be safe.

The test uses the Linux AP as the discriminator, which needed a new orchestrator branch: _bring_up_linux now handles a hostapd-ap role (→ linux_peer.bring_up_ap), with the build-var / dry-run peer lines guarded so a hostapd-ap STA (associates by SSID) doesn't get the mesh-interop MAC/IP. PASS end-to-end through the harness.

#10 · Multi-STA TWT: both INSTALLED (T2 multi-twt)

Two STAs concurrently negotiating TWT against one ESP SoftAP — the "2 authorized STAs" concurrency the single-STA twt test asserts but never runs. No new firmware (reuses regtest-twt-sta on board1 + board2). The work was a multi-reporter harness extension: _run_orchestrated now collects every reporter=True role, captures each (the first stays associated while the next joins, so both coexist), and _record_multi_verdict ANDs the verdicts. The single-reporter path was preserved (re-ran twt = PASS). Verified: 2/2 reporters INSTALLED concurrently.

#9 · Deep-sleep duty-cycle reconnect (dscycle)

Over N cycles, prove the battery-leaf rejoins on every wake — counting both RECONNECTED lines and the long port-gone gaps that prove genuine deep-sleep. Structural gate, no PPK2 measurement.

Two subtleties, handled honestly

(a) Re-opening the flapping native USB-JTAG resets the just-woken board, so the firmware's wake_cause reads UNDEFINED — so deep-sleep is proven from the port-gone duration (>4 s = a real deep-sleep wait; a bare reset returns in ~2 s), which the reset can't fake. (b) The sleep app wedges the USB; reliable recovery is a 5 s PPK2 power-off (2.5 s leaves board2 dark) + a tight esptool flash in the fresh-boot window.

◐ implemented + logic-verified; a clean PASS needs a healthy board2

First run: a clean 2 reconnects + 2 consistent deep-sleep gaps (logic verified). Later runs on the repeatedly-wedged, over-stressed board2 landed INCONCLUSIVE (2/3) — the wakes went flaky and the gate correctly returned INCONCLUSIVE, never a false PASS (the suite doctrine working). The test is code-complete and the recovery works; a 3/3 PASS needs a rested board2 + reliable C6 wakes.

#8 · HOST_LIGHT_SLEEP ladder variant (tp --light-sleep)

HOST_LIGHT_SLEEP is now a build-arg (#ifndef guard + the Makefile threading HOST_LIGHT_SLEEP=1 → -D REGTEST_HOST_LIGHT_SLEEP → the CMake define). tp --light-sleep builds it and scores against POWER_BANDS_LS, calibrated on-rig (3 runs): LS Dyn-PS ~25.8 mA / WNM ~22–25 mA, tight, both APs near-identical. Bands dyn-ps (35, 48) / wnm (35, 46); fresh run PASS, gross regression FAIL, grey INCONCLUSIVE (synthetic-verified); the host-awake bands are untouched.

Finding — the "stronger gate" advantage is masked on this rig

In the clean reference (TX-capped AP), light-sleep backfires WNM 1.17.8 ~4 mA → 1.17.9 ~30 mA (~7×). But on our RX-overloaded rig the LS 1.17.8 already reads ~25.8 mA, so the multiple vs a ~30 mA 1.17.9 is no bigger than the host-awake gate. The variant works + is tight/repeatable, but only becomes the intended stronger gate under v2 (cap the AP TX) — the same limitation the main tp tier has.

Net + follow-ups

All five in the (already-uncommitted) regtest harness; nothing committed. T0 stays green — fw-blob PASS, drift clean, the new/changed apps build. Follow-ups: v2 (cap the AP TX) unlocks the reference-like numbers that make both the main tp and the LS gate genuinely stronger; and a clean 3/3 dscycle on a rested board2.

Bench radio-silent after (3 ESPs on rimba-hello, chronite wlan1 down, board2 held by tools/ppk2_hold.py).
