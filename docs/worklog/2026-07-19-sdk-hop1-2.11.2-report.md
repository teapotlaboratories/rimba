# SDK Hop 1 — morselib 2.10.4 → 2.11.2

**Date:** 2026-07-19 · **Scope:** `components/halow` submodule + `firmware/test-mesh-ap-gate` · **Status:** functionally complete, T0 + T2 green, **uncommitted** (tag `2.11.2-esp32-1` is the owner's call)

> This is **Hop 1** of the staged forward-port that brings MorseMicro's upstream `mm-iot-sdk` **into** our
> self-contained vendored tree — upgrading the bundled **morselib from 2.10.4 → 2.11.2** while keeping our
> 24 PRs of work (IBSS, 802.11s mesh + SAE/AMPE/CCMP, mesh-gate/AP concurrency, TWT, SW-CCMP bulk-AES,
> A-MPDU, real-RC airtime). Hop 2 (2.11.2 → 2.12.3) is the additive follow-up.

---

## TL;DR

The hop is **three layers stacked on one commit**:

1. **The upstream SDK forward-port (2.10.4 → 2.11.2).** Upstream refactored the datapath from a *single global
   ops table* to a *per-VIF ops model*. This is the load-bearing change and it touches exactly the files our
   features live in.
2. **Re-architecting our features onto the new model.** We deleted our custom mesh-gateway machinery and
   re-expressed mesh + AP concurrency natively on upstream's two-slot per-VIF design (new `umac_datapath_mesh.c`
   + `umac_datapath_ibss.c`, a rebuilt `umac_interface.c`).
3. **Six fixes** found by on-air regression testing — none a re-arch *logic* bug; each is either our own ESP
   glue or a latent assumption the new per-VIF model exposed.

The chip firmware is **1.17.8 throughout** — no version changed in this hop. The whole bench (3 ESP + 4 Linux
nodes) runs the *identical* `mm6108.bin` binary, single-sourced from `vendor/morse-firmware` and regenerated to
the ESP-loadable `.mbin` at build time (see §3).

**Result:** every on-air capability that worked on 2.10.4 works on 2.11.2 — STA, IBSS, TWT, multi-TWT,
SW-CCMP mesh, A-MPDU, mesh peering, mesh relay, AP↔STA, the concurrent **mesh-gate**, **and ESP↔Linux mesh
interop** — all verified green.

## 1 · The architectural shift: global ops → per-VIF ops

The single most important thing this hop adds is upstream's **per-VIF datapath model**. On 2.10.4 the datapath
had **one global ops table** (`data->ops`) and stamped a single active `data->vif_id`. On 2.11.2 each interface
carries **its own ops table**, TX frames are stamped with a `tx_metadata->vif_id`, and RX frames are demuxed by
the firmware's rx vif id.

### Before (2.10.4) — one global ops table serving whatever was active

```c
/* umac_datapath_data.h (2.10.4) */
struct umac_datapath_data {
    const struct umac_datapath_ops *ops;   /* ONE table, swapped per active mode */
    uint16_t vif_id;                       /* ONE active vif                     */
    ...
};

/* TX: 5-arg, vif implied by the single global table */
enum mmwlan_status umac_datapath_tx_frame(struct umac_data *umacd,
                                          struct mmpkt *pkt,
                                          enum encryption_mode enc,
                                          const uint8_t *ra,
                                          uint16_t vif_id);   /* <- passed in */
```

### After (2.11.2) — a two-slot table, ops resolved per VIF

```c
/* umac_interface.c (2.11.2) — two fixed slots */
struct umac_interface_data {
    struct umac_interface_vif_data data_sta;   /* VIF_STA slot (STA / MESH / IBSS live here) */
    struct umac_interface_vif_data data_ap;    /* VIF_AP  slot (concurrent SoftAP)           */
};
/* each slot stores its own ops + fw vif id */
struct umac_interface_vif_data {
    const struct umac_datapath_ops *datapath_ops;
    uint16_t vif_id;
    ...
};

/* TX: 4-arg — vif comes from the frame's own pre-stamped metadata */
enum mmwlan_status umac_datapath_tx_frame(struct umac_data *umacd,
                                          struct mmpkt *pkt,
                                          enum encryption_mode enc,
                                          const uint8_t *ra);  /* vif via tx_metadata->vif_id */
```

### The API signatures that changed (and every caller we had to adapt)

| Symbol | 2.10.4 | 2.11.2 | why it matters to us |
|---|---|---|---|
| `umac_interface_add()` | `(umacd, type, bssid)` → returns vif_id | `(umacd, type, datapath_ops, bssid)` | every add-interface call now supplies its ops table |
| `umac_datapath_tx_frame()` | 5-arg (`…, vif_id`) | 4-arg (vif via `tx_metadata->vif_id`) | all TX callers re-stamped |
| `morse_cmd_tx()` | 5-arg, `int` return | 6-arg (`…, skip_errlog`), `enum mmwlan_status` | our mesh driver cmds add `,false` + enum |
| `mmdrv_add_if / rm_if / update_sta_state` | `int` | `enum mmwlan_status` | return-type propagation |
| `umac_datapath_ops` struct | `set_stad_sleep_state` … | `+update_stad_state_rx`, `+handle_frame_unknown_sta`, `+supp_l2_sock_receive`, `+type` | our ops tables re-shaped to match |
| offload / standby / whitelist APIs (`mmwlan.h`) | present | **removed** | zero source usage — only docs referenced them |
| `driver_health.c` | present | **moved** → `umac/health_check/umac_health_check.c` | CMake SRCS re-synced |

The RX side changed to match: instead of "whatever the global ops is," a received frame is routed to the slot
for its firmware-stamped vif id:

```c
/* umac_datapath.c (2.11.2) — RX resolves ops by the FW-stamped vif */
uint16_t vif_id = umac_interface_get_vif_id_from_rx_metadata(metadata);
const struct umac_datapath_ops *datapath_ops =
    umac_interface_get_datapath_ops_by_vif_id(umacd, vif_id);
```

## 2 · Re-architecting our mesh-gate onto the per-VIF model

Our biggest feature — the **all-ESP mesh-gate** (mesh primary + a concurrent SoftAP on one MM6108, with L3
routing between them) — was built on the old global-ops model with a bespoke **gateway ops table** that served
*both* the mesh stad and the AP client. That model no longer exists on 2.11.2.

The owner's call was **"re-arch fully clean — delete the fallbacks, trust the FW vif tagging."** So:

**Deleted (our old gateway machinery):**

```
- datapath_ops_gateway          (the one table that served mesh + AP together)
- gw_*                          (gateway helpers)
- configure_gateway_mode()      (mode switch)
- data->ap_vif_id               (the second-vif tag on the global struct)
```

**Created (native per-VIF ops tables):**

| File | Lines | Ops table | Slot |
|---|---|---|---|
| `umac_datapath_mesh.c` | 250 | `umac_datapath_ops_mesh` | VIF_STA |
| `umac_datapath_ibss.c` | 191 | `umac_datapath_ops_ibss` | VIF_STA |
| `umac_datapath_sta.c` *(upstream)* | 227 | `umac_datapath_ops_sta` | VIF_STA |

The per-VIF model **natively subsumes the gateway**: MESH registers on the **VIF_STA** slot, the concurrent AP
on the **VIF_AP** slot, RX demuxes by the FW rx vif, TX by the stamped `tx_metadata->vif_id`, and the dequeue
drains *both* slots. No custom "gateway" needed.

The TX path picks the slot from the frame's requested VIF (this is where fixes #3/#5 later plug in):

```c
/* umac.c mmwlan_tx_pkt (2.11.2) — resolve the vif slot from metadata.vif */
if (metadata.vif == MMWLAN_VIF_STA) {
    vif_id = umac_interface_get_vif_id(umacd, UMAC_INTERFACE_STA);
    if (vif_id == MMDRV_VIF_ID_INVALID)
        vif_id = umac_interface_get_vif_id(umacd, UMAC_INTERFACE_MESH); /* mesh lives on the STA slot */
} else if (metadata.vif == MMWLAN_VIF_AP) {
    vif_id = umac_interface_get_vif_id(umacd, UMAC_INTERFACE_AP);
} else /* UNSPECIFIED */ {
    /* infer — but ambiguous when BOTH a STA and an AP vif are valid (see fixes #3/#5) */
    ...
}
```

## 3 · Firmware: uniformly 1.17.8 (no version change this hop)

**Correction to record:** an earlier draft framed this hop as a "1.17.6 → 1.17.8 firmware-parity bump." That
was wrong. The bench — all 3 ESP nodes and all 4 Linux nodes — runs the *identical* chip firmware **1.17.8**
(`mm6108.bin`, 480664 B, md5 `cfe56db2`, `rel_1_17_8`), and it has for the whole HaLow effort.

The reason it's identical is the build system, not luck. `cmake/mm-fw-gen/` treats **`vendor/morse-firmware`
as the single source of truth** and *regenerates* the ESP-loadable Morse-TLV `.mbin` from that ELF **at build
time** (`convert-bin-to-mbin.py`), then objcopy-wraps it in. Per the generator's own header comment:

> *"Nothing firmware-related is committed under `components/`; the firmware version is whatever
> `vendor/morse-firmware` is pinned at."*

So the `mm6108.mbin` **committed inside the morselib submodule** (`components/mm-iot-sdk/framework/morsefirmware/`)
is **never used by the build** — it's a dormant artifact of the vendored SDK package. I did bump it
(399468 → 399796 B) to 2.11.2's bundled blob for tree-fidelity, but that change has **zero build effect**: the
ESP loads the same 1.17.8 binary the Linux nodes do, regenerated from `vendor/morse-firmware`, before and after.

⇒ **There was no firmware version to reconcile.** Fix #2 below is *not* a "1.17.6 code meeting 1.17.8" issue —
it's a mesh-vif setup difference in the 2.11.2 forward-port tripping a quirk on the (unchanged) 1.17.8 firmware.

## 4 · The fixes (the debugging story)

Regression testing on 2.11.2 surfaced five real breaks. The through-line: **the re-arch's *structure* was
sound** — the bugs were latent assumptions in our glue and one-per-VIF-model corners the global model had
masked, plus one mesh-vif/beacon-config quirk on the (unchanged) 1.17.8 firmware that the 2.11.2 mesh-vif
setup surfaced.

### Fix 1 · `get_tx_seq` — a key *type* passed where a key *id* was expected

Upstream moved the `key_type → key_id` resolution out of `connection_keys`, but our resolution kept the enum.
Result: a key **type** flowed into a slot that wanted a key **id** — corrupting the mesh SW-CCMP packet number.

```diff
  /* umac_keys.h / umac_keys.c */
- uint64_t umac_keys_get_tx_seq(struct umac_sta_data *stad, enum umac_key_type key_type);
+ uint64_t umac_keys_get_tx_seq(struct umac_sta_data *stad, uint8_t key_id);

  /* umac_datapath.c (caller) */
- seq = umac_keys_get_tx_seq(stad, key_type);
+ seq = umac_keys_get_tx_seq(stad, (uint8_t)key_id);   /* MM_UNUSED(key_type) in sw_ccmp_encrypt */
```

### Fix 2 · Mesh vif wouldn't start — fw 1.17.8 rejects a standalone beacon config

The mesh vif failed at `mmwlan_mesh_start` with the firmware returning `fw_status=14` to a standalone
`BSS_BEACON_CONFIG`. The firmware is 1.17.8 the whole time (§3) — what changed is the **2.11.2 mesh-vif
bring-up**, which trips a rejection the 2.10.4 path didn't. The Linux 1.17.8 driver confirms the shape of the
fix: it does **not** send `config_beacon_timer` on the initial mesh start (the later
`MESH_CONFIG(enable_beaconing)` arms beaconing). Following that reference, we made the call **non-fatal**.

```diff
  /* umac_mesh.c — mmwlan_mesh_start */
  ret = mmdrv_config_beacon_timer(vif_id, true);
- if (ret != 0) { goto fail; }
+ if (ret != 0) {
+     /* fw 1.17.8 rejects a standalone BSS_BEACON_CONFIG on a mesh vif (fw_status 14); the Linux 1.17.8
+      * driver does NOT send it on the INITIAL mesh start — MESH_CONFIG(enable_beaconing) arms beaconing.
+      * Treat it as non-fatal. */
+     MMLOG_ERR("MESH: config_beacon_timer fw_status=%d (non-fatal on 1.17.8)\n", ret);
+ }
```

**Verified:** mesh vif up + beaconing → mesh-peering, mesh-relay, ampdu-cap all PASS.

### Fix 3 · AP SoftAP: 0 ICMP replies — egress left the VIF UNSPECIFIED

A STA associated to our SoftAP and completed the 4-way, but **every** ICMP reply died at
`mmwlan_tx_pkt: Unable to infer VIF ID`. Root cause is in **our ESP glue** (`mmhalow.c`), not the SDK:
`halow_transmit` initialised the TX metadata with only `.tid`, leaving `.vif = 0 = MMWLAN_VIF_UNSPECIFIED`.
Under the old global model, UNSPECIFIED resolved via the single `data->vif_id`; under the per-VIF model the
inference **bails when both a STA vif and an AP vif are valid** — and mwlan *always* boots a STA vif, so a plain
AP app (AP = vif 1) is inherently ambiguous.

```diff
  /* components/halow/mmhalow.c — halow_transmit */
+ mmhalow_netif_driver_t *driver = (mmhalow_netif_driver_t *)h;
  struct mmwlan_tx_metadata metadata = {
      .tid = 0,
+     .vif = driver->tx_vif,      /* was implicitly MMWLAN_VIF_UNSPECIFIED */
  };

  /* mmhalow.h — new per-driver field (calloc-default UNSPECIFIED, so STA/mesh keep inferring) */
+ enum mmwlan_vif tx_vif;

  /* mmhalow.c — mmhalow_wifi_start (the AP path) */
+ morse_drv->tx_vif = MMWLAN_VIF_AP;   /* tag AP egress so the per-VIF datapath routes it correctly */
  mmwlan_ap_enable(&morse_drv->ap_args);
```

**Verified:** `ap-sta-ping` → 15/15 ICMP replies (was 0/15).

### Fix 4 · Mesh-gate: a new STA can't associate — the reassign LUT only ran on INVALID

With mesh + AP concurrent, a client's **broadcast PROBE_REQ** is tagged by the firmware with the **mesh** vif
(a *valid* id, not INVALID). The re-arch's frame-type→interface reassignment LUT (`PROBE_REQ → AP`) only ran
when the FW vif was INVALID, so the probe was handled by the mesh ops and **the AP never answered** — no
association. (Instrumentation showed `PROBE_REQ vif=0`; after the fix, `probe/auth/assoc/action all vif=1=AP`.)

The fix makes the LUT **authoritative**: these frame types have a fixed owner regardless of the FW tag, and the
reassign only overrides when the target interface exists — so it's a **no-op** for mesh-only / STA-only /
plain-AP.

```diff
  /* umac_datapath.c — umac_datapath_rx_frame_preprocess */
- if (vif_id == MMDRV_VIF_ID_INVALID) {
-     static const struct umac_datapath_vif_reassignment lut[] = { {PROBE_REQ, AP}, {S1G_BEACON, STA}, {PROBE_RSP, STA} };
-     umac_datapath_reassign_vif_id(umacd, metadata, fvts, lut, N);
-     vif_id = umac_interface_get_vif_id_from_rx_metadata(metadata);
- }
+ /* authoritative: applied REGARDLESS of the FW rx-vif tag (a concurrent mesh 'claims' broadcast RX) */
+ static const struct umac_datapath_vif_reassignment lut[] = { {PROBE_REQ, AP}, {S1G_BEACON, STA}, {PROBE_RSP, STA} };
+ bool lut_managed = umac_datapath_reassign_vif_id(umacd, metadata, fvts, lut, N);   /* now returns bool */
+ vif_id = umac_interface_get_vif_id_from_rx_metadata(metadata);
+ if (vif_id == MMDRV_VIF_ID_INVALID && !lut_managed)
+     MMLOG_WRN("RX VIF unknown, frame type %x:%x\n", ...);   /* warn lifted out of reassign */
```

`umac_datapath_reassign_vif_id` now `return`s `true`/`false` (matched a LUT row?) instead of `void`, and only
mutates `metadata->vif_id` when the target interface exists — hence safe everywhere.

**Verified:** the STA associates to the concurrent gate AP (mesh-peering still PASS = no mesh regression).

### Fix 5 · Mesh-gate: forwarding STA → mesh was dead — the gate's mesh netif egressed UNSPECIFIED

With association fixed, the STA associated but got **0/15 replies from the far mesh node** — the
`STA → AP → ip_forward → mesh → far-node` path didn't deliver. The gate's mesh netif is mmhalow's default
(`g_mesh_netif = mmhalow_get_netif()`), and the gate calls `mmwlan_mesh_start` **directly** (not
`mmhalow_wifi_start`), so its `tx_vif` stayed UNSPECIFIED — ambiguous once the concurrent AP vif exists, so the
mesh-bound forward was dropped/misrouted.

Fix: a small new mmhalow API lets a caller tag its netif's egress VIF; the gate tags its mesh netif **STA**
(the mesh lives on the STA host-slot; morselib resolves STA → MESH via the TX fallback).

```diff
  /* components/halow/mmhalow.h + .c — new API */
+ void mmhalow_set_tx_vif(enum mmwlan_vif vif);   /* sets driver->tx_vif */

  /* firmware/test-mesh-ap-gate/main/app_main.c */
  g_mesh_netif = mmhalow_get_netif();
+ /* this netif egresses on the MESH vif (STA host-slot) — tag it so gateway-forwarded packets go out the
+  * mesh vif; UNSPECIFIED is ambiguous once the concurrent AP vif comes up. */
+ mmhalow_set_tx_vif(MMWLAN_VIF_STA);
```

**Verified:** `mesh-ap` PASS (full STA→far-node ping) and `mesh-ap-multi-twt` PASS.

### Fix 6 · `twt-assoc` regression — the assoc-time TWT seed was compiled out by a `#if` guard

**Correction to the record:** the first draft of this report listed `twt-assoc`'s INCONCLUSIVE as a
"pre-existing accessor limitation, not a regression." That was **wrong**. Checked against the results history,
`twt-assoc` **PASSED on 2.10.4** (2026-07-17) and only INCO'd on 2.11.2 — a genuine Hop-1 **regression**.

The hostap 2.11.2 forward-port tightened the assoc-time TWT-seed guard in `wpa_supplicant.c` (two sites):

```diff
- #if defined(CONFIG_S1G_TWT) && defined(CONFIG_DRIVER_NL80211_MORSE)
+ #if defined(CONFIG_S1G_TWT) && (defined(CONFIG_DRIVER_NL80211_MORSE) || defined(MM_IOT))
```

The ESP build uses the `mmwlan` driver and defines **`MM_IOT`**, *not* `CONFIG_DRIVER_NL80211_MORSE` — so the
whole assoc-time TWT-config block **compiled out**: the `.twt_conf` driver op (`mmwpas_twt_conf`) never fired →
no agreement was seeded → the assoc request carried no TWT IE → nothing to install → INCONCLUSIVE. It **compiles
fine either way**, which is precisely why compile-verify couldn't catch it — the "268-file hostap update = bench
risk" the plan flagged, materialised. The fix restores the codebase's own idiom (`driver.h:32`).

**Verified** with serial probes (the `.twt_conf` op fires → agreement seeded → assoc-req IE emitted → AP grants
→ INSTALLED on flow 0) and a harness `twt-assoc` **PASS**. This takes the full T2 to **12 PASS / 0 INCONCLUSIVE**.
